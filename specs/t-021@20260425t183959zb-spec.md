# Specification: T-021@20260425T183959Zb

## Problem Statement

Sub-task `T-021@20260425T183959Za` introduced a combined sampler plan+playback
status line via the opt-in `combine_activity_and_playback=True` path on
`senseweave.sample_status.sample_status_text(...)`. That combined wording is
already wired into `senseweave.operator_diagnostics`, but the operator-facing
**face display renderer** at `my-claw/tools/face_display.py` still calls
`sample_status_text(...)` in its legacy either/or mode. As a result the small
face monitor only shows half of the sampler story when the sampler plan and
sampler playback states are both active.

The face display should mirror operator diagnostics so operators see the same
combined sampler line on both surfaces.

## Technical Approach

1. Add a thin helper `face_display_sample_status_text(activity, playback, monitor)`
   to `my-claw/tools/senseweave/sample_status.py`. The helper is a one-call
   wrapper around `sample_status_text(...)` with
   `combine_activity_and_playback=True` so the face display and operator
   diagnostics share the same opt-in seam without duplicating combine logic.
2. Update `my-claw/tools/face_display.py` to import the new helper (with the
   same `try/except ImportError` fallback pattern already used for
   `sample_status_text`) and invoke it in the existing sample-layer status
   block (around line 380), preserving the existing font, color offset, and
   centered y=170 placement so the line continues to mirror the surrounding
   diagnostic field formatting.
3. Keep all degradation behavior unchanged: missing
   `/tmp/sample_dsp_activity.json`, missing `/tmp/sample_playback_state.json`,
   missing `/tmp/self_listen.json`, corrupt JSON, and an empty status string
   must continue to be silently skipped without crashing the renderer.
4. Do not change the legacy default behavior of `sample_status_text(...)` —
   only the new helper opts into combined wording, matching the
   already-shipped pattern from sub-task Za.

## Edge Cases

- When `playback["playing"]` is `False` and an activity plan is present, the
  combined helper must still emit the plan-only phrase (e.g.
  `currently sampling theramini in via room mic · grain cloud`) so the face
  does not go silent during plan-only periods.
- When both states are present and playback is active, the helper must emit
  both clauses joined by ` · ` (e.g.
  `currently sampling theramini in via room mic · playing sample freeze bed
  from source self bus`), matching the wording already verified for
  operator diagnostics.
- When monitor state reports `error: no_capture`, the helper must surface
  `monitor offline` first, preserving the degradation contract from
  `sample_status_text`.
- When `activity` is empty or `None`, the helper must not crash and should
  return only the playback phrase if playback is active, or an empty string
  otherwise.
- When all three inputs are empty/missing, the helper must return an empty
  string so the face renderer's existing `if sample_text:` guard skips the
  draw call cleanly.

## Assumptions

- No new dependencies, database columns, or migrations are required — this
  task is purely diagnostics-formatting wiring.
- Hardening items in the task brief about `bootstrap_identity()` startup
  invocation, identity persistence integration tests, and standalone vs
  federated runtime modes belong to the identity/runtime workstream, not to
  this face-display formatting task. They are recorded as out-of-scope in
  ESCALATIONS.md if the verifier flags them.
- Tests cover the helper directly (which is the only new logic in this
  task). The face display renderer itself is a pygame/PIL surface and is
  exercised at runtime; importing it under headless pytest pulls heavy
  graphics dependencies and is intentionally not unit-tested here. The
  call-site change is verified by source-level assertion that the renderer
  imports and invokes the new helper.
