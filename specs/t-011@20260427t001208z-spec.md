# Spec: T-011@20260427T001208Z

## Problem Statement

The generation backend (Replicate / Modal / local) has accreted significant
operational surface area: USD budget caps (T-031, T-032), an SQLite-backed
queue, a content-addressed cache, an IDyOM long-term-model collapse audit, a
sensory journal, and date-partitioned audio storage. Operators currently have
to read source code in `my-claw/tools/senseweave/generation/` to diagnose
incidents. There is no central runbook that names the on-disk paths, env
vars, and recovery procedures, so first-boot incident response is slower than
it needs to be and tribal-knowledge dependent.

## Technical Approach

1. Create a new runbook file at `docs/runbooks/generation-backend.md`
   following the established style of `docs/r750-bare-metal-runbook.md`
   (H1 title, parts/sections, callouts, tables for env vars and paths,
   bash blocks with expected output).
2. Source all facts from the live code: budget caps from
   `my-claw/tools/senseweave/generation/budget.py`, queue states from
   `queue.py`, cache from `cache.py`, audit from `health.py`, audio
   storage from `storage.py`, journal path from
   `senseweave/sensory_journal.py`. Cross-link to the existing
   `sdp/telemetry/cost-model.md`.
3. Cover every operator workflow named in the acceptance criteria:
   status check, widen/tighten budget cap, backend switch, IDyOM LTM
   rollback, queue unblock, cache inspection, on-disk locations.
4. Document every env var the generation system actually reads
   (`CYPHERCLAW_GENERATION_DAILY_CAP_USD`, `CYPHERCLAW_GENERATION_MONTHLY_CAP_USD`,
   `IDYOM_KL_AUDIT_CONFIG`, `REPLICATE_API_TOKEN`, `MODAL_TOKEN_ID`,
   `MODAL_TOKEN_SECRET`) and its default if any.
5. Include exactly five worked debugging scenarios that exercise the
   covered surfaces (cap breach, stuck queue, cold cache, collapse alert,
   backend outage failover).
6. Lock the runbook against drift via a static regression test in
   `tests/test_runbook_generation_backend.py` modeled on the existing
   `tests/test_cypherclaw_sampler_artistic_intent.py` pattern.

## Edge Cases

- The `local` backend (`LocalAdaClient`) raises `NotImplementedError` on
  every call until a GPU is installed; the backend-switch section must
  document this rather than imply it is functional.
- Daily/monthly caps roll over at UTC midnight; the runbook must say UTC
  so operators in PT do not miscalculate when a cap will reset.
- IDyOM collapse audit never auto-rollbacks; rollback is operator-driven.
- Queue items in `running` state past their 120 s client timeout are
  candidates for manual clearance, but the queue does not auto-detect
  this.
- Cache eviction is LRU by `accessed_at`; manually deleting cache files
  without touching `cache_index.json` will leave a stale index entry.
