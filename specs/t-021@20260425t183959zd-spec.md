# Specification: T-021@20260425T183959Zd

## Problem Statement

Sub-tasks `T-021@20260425T183959Za`–`Zc` landed the combined sampler
plan+playback wording, threaded it through the face display, and added
an inkplate renderer that consumes the same helper. The shared formatter
`senseweave.sample_status.sample_status_text(..., combine_activity_and_playback=True)`
is now the single source of truth for the sampler line on every operator
surface, including the `sample_source` field on
`senseweave.operator_diagnostics.OperatorDiagnostics`.

The existing `tests/test_operator_diagnostics.py` exercises a handful of
realistic surface scenarios but does not pin the four discrete states of
the operator-diagnostics sampler line on its own:

1. **sampling-only** — capture is active but playback is silent.
2. **playing-only** — playback is active but capture is idle.
3. **both** — capture and playback are both active.
4. **neither** — capture and playback are both idle.

Without dedicated state-by-state coverage, a future change to the shared
formatter, the operator-diagnostics wiring, or the composer-state
fallback could quietly drop one of these four shapes from the
operator-facing status line and slip past CI.

## Technical Approach

1. Add a new test module `tests/test_operator_diagnostics_sample_status.py`
   focused exclusively on the `OperatorDiagnostics.sample_source` field
   produced by `collect_operator_diagnostics(...)`. Keeping it in its own
   file mirrors the per-surface naming used for
   `tests/test_face_display_sample_status.py` and
   `tests/test_sample_status.py`, and avoids extending the already
   large `test_operator_diagnostics.py`.
2. Drive the function through its public seam (`DiagnosticPaths` +
   `collect_operator_diagnostics`) using `tmp_path` JSON fixtures, the
   same pattern the existing operator-diagnostics tests use, instead of
   reaching into the private `_sample_source_status` helper. This keeps
   the test resilient to refactors that move the formatting around as
   long as the operator-facing field stays correct.
3. For each of the four states, write a single small focused test that
   pins the wording the operators see today:
   - sampling-only → `currently sampling …` and **no** `playing sample …`
     fragment.
   - playing-only → `playing sample …` and **no** `currently sampling …`
     fragment.
   - both → both fragments, joined by ` · ` (the shared helper's
     separator).
   - neither → falls back to `unavailable` when no composer-state
     `sample_source` is present.
4. Reuse the existing `_write_json` helper pattern from
   `tests/test_operator_diagnostics.py` and avoid touching any
   production code; the implementation already supports all four states.

## Edge Cases

- The neither-state must still degrade to `unavailable` rather than an
  empty string when both authority files are missing or empty, because
  the daemon serializes `sample_source` straight into status payloads.
- The sampling-only state must require `capture_ready=True` and
  `trigger_now=True` to render the `currently sampling …` wording; the
  test pins the human-facing phrase rather than the underlying flags so
  a wording change is caught immediately.
- The playing-only state must not surface a `currently sampling …`
  fragment even if the activity file is present but inactive
  (`capture_ready=False`, `trigger_now=False`), so that "holding" plans
  do not bleed into the playback-only line.
- The both-state assertion uses ` · ` as the joiner because that is the
  separator the shared helper emits today; if the helper ever changes,
  the test will surface the regression.
- This task is test-only; it does not modify production code, add new
  dependencies, or alter any other surface.

## Assumptions

- No new dependencies or migrations are required; this task only adds
  a new test file under `tests/`.
- The four-state contract is already satisfied by the current
  implementation of `_sample_source_status` and the shared
  `sample_status_text(..., combine_activity_and_playback=True)` helper,
  so the new tests should pass on first run against `main`. They are
  additive regression coverage rather than a TDD red-then-green driver
  for new behavior — this is the appropriate shape because the task
  brief explicitly asks for "unit tests covering" the existing four
  states.
- The auto-generated hardening anchors about `bootstrap_identity()`
  startup wiring are unrelated to this test-only task; they are already
  covered by the existing identity, governor, and self-critique suites
  and will be re-validated by the standard `pytest tests/ -x` run
  required by the validation block.
- `progress.md` is generated and already dirty with unrelated queue
  state, so only the single task line will be edited manually.
