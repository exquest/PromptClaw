# Specification: T-013@20260425T183959Zb

## Problem Statement

The CypherClaw sampler subsystem has now landed in pieces — a capture daemon
with rolling buffers and tag extraction, a buffer pool, an OSC dispatcher,
a dedicated effects bus, and two render-time anti-pattern checks — but no
single document explains how those pieces fit together. The PRD at
`my-claw/sdp/prd-cypherclaw-sampler.md` describes the *intent* and the
phased plan, and the musicianship roadmap (updated in T-013@20260425T183959Za)
records *status*, but there is no architectural reference that an engineer
or new collaborator can read to understand:

- the components that exist today and where they live,
- how a captured moment travels from the room mic to a granular cloud in
  the master bus,
- the misuse patterns the system actively guards against, and
- the integration points where the sampler plugs into the rest of the
  CypherClaw runtime.

CCS-033 in the sampler PRD calls for exactly this document at
`docs/cypherclaw-sampler-architecture.md`. This subtask delivers it.

## Technical Approach

1. Create `docs/cypherclaw-sampler-architecture.md` as a focused
   architectural reference (not a re-statement of the PRD or roadmap).
2. Structure the document so that an engineer can navigate to the
   information they need without re-reading the whole thing:
   - Overview of what the sampler is and is not.
   - Components: list each landed module with its responsibility and
     where it lives in the tree.
   - Signal flow: a single end-to-end diagram-in-prose tracing the path
     from JACK input → capture → tag → store → select → dispatch →
     synth → effects → master bus.
   - Antipatterns guarded against: name each detector, what it watches
     for, and how it surfaces in the metrics gate.
   - Integration points: identity, cast planner, mix engine, composer
     hooks, and operator diagnostics.
   - Status delta: a short note that the doc reflects the *current
     partial-rollout* state and links back to the roadmap and PRD for
     the longer plan.
3. Reference concrete file paths so the doc is verifiable against the
   tree and so future readers can jump straight to the source.
4. Cross-link the existing musicianship roadmap and sampler PRD; do not
   duplicate their content.
5. Add a focused regression test
   (`tests/test_cypherclaw_sampler_architecture.py`) that pins the
   document's required sections, the named landed components, the named
   anti-patterns, and the cross-links so future edits cannot silently
   drop the architectural contract.

## Edge Cases

- The document must reflect *current* tree state, not the full PRD
  vision. Pieces such as `SampleLibrary`, `SampleSelector`, and
  `sw_sampler.scd` are still absent and must be flagged as pending —
  not described as if they exist.
- Anti-pattern names must match the live identifiers in
  `render/antipatterns.py` (`sampler_dominating`,
  `sampler_silent_quintet_member`); a regression test pins both names so
  a rename anywhere drives the doc to update with the code.
- Component file paths used in the doc must actually exist in the tree;
  the regression test verifies each referenced path so the doc cannot
  rot into pointing at moved or deleted modules.
- The doc must not silently re-define the quintet differently from the
  roadmap — it should reference the roadmap's snapshot rather than
  inventing a new authoritative list.
- This task is documentation-only. It must not modify runtime code,
  introduce migrations, or add new dependencies.

## Assumptions

- "Sampler architecture" for this subtask means the architecture as it
  stands in the current tree, plus an explicit pending-work pointer.
  A full forward-looking architecture covering the unbuilt
  `SampleLibrary` / `SampleSelector` / `sw_sampler.scd` belongs to
  later tasks once those modules land.
- The artistic-intent document (`docs/cypherclaw-sampler-artistic-intent.md`)
  is a sibling subtask under CCS-033 and is out of scope here.
- The auto-generated startup hardening checks
  (`bootstrap_identity` invocation, `FirstBootAnnouncer` sequencing)
  describe runtime startup, not documentation. They are already covered
  by `tests/test_first_boot.py` and `tests/test_governor_integration.py`,
  and the `sample_capture_daemon.py` `main()` entry point already calls
  `bootstrap_identity()`. This documentation task does not touch those
  paths and an out-of-scope note will be filed in `ESCALATIONS.md`.
- No new dependencies and no migrations are required.
