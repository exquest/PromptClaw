# T-022@20260515T214233Z Specification

## Problem Statement

PAL deployment work needs a repo-managed source of truth for the files that are
intended to live under `/opt/pal` on the Phase 1 host. Today the PRD and runbook
name the live layout, and `pal-2026/ops/templates/` contains several source
templates, but there is no structured manifest that future deploy diff, plan,
backup, apply, or rollback code can consume.

This task implements PAL-027 from
`sdp/prd-pal-2026-agentic-ops-platform.md`: create the manifest only. It must be
local, inspectable, deterministic, and must not include provider secrets,
credential paths, API tokens, Tailscale auth keys, or runtime log contents.

## Technical Approach

- Add a JSON deployment manifest at `pal-2026/ops/deployment-manifest.json`.
- Add a small typed stdlib-only loader/validator in `promptclaw/pal_deploy.py`
  so tests and later deploy tooling do not hand-roll manifest parsing.
- Model each file entry with:
  - repo-relative `source`
  - absolute `/opt/pal/...` `target`
  - octal string `mode`
  - `owner`, `group`
  - `kind`
  - `service_impact`
  - `required`
- Include the current host-managed PAL files as first-class manifest entries:
  - `/opt/pal/scripts/start_all.sh`
  - `/opt/pal/scripts/start_ollama.sh`
  - `/opt/pal/scripts/start_router.sh`
  - `/opt/pal/scripts/auto_shutdown.sh`
  - `/opt/pal/config/shutdown.conf`
  - `/opt/pal/router/app.py`
  - `/opt/pal/DEPLOYMENT_INFO.md`
- Keep Docker fallback templates represented but not authoritative:
  - `/opt/pal/router/Dockerfile`
  - `/opt/pal/docker-compose.yml`
- Add repo-managed host script templates only if needed to make every required
  manifest source path real and deployable.
- Keep runtime data out of the file entries. Directories such as logs and
  Ollama model storage can be listed as runtime directories, not files to sync.
- Do not add CLI commands, remote SSH behavior, deploy planning, backups,
  apply, rollback, dependencies, migrations, database columns, or secrets.

## Edge Cases

- A manifest entry whose target is not under `/opt/pal/` must be rejected.
- Duplicate target paths must be rejected so deploy diff/apply cannot race two
  sources into the same remote file.
- Required manifest sources must exist in the repository.
- Runtime paths such as `/opt/pal/logs/*.log`, `/opt/pal/ollama/**`, and
  `/opt/pal/config/override.flag` must not be treated as managed files.
- Manifest text must not include secret material, secret environment variable
  names, private key paths, Tailscale auth key prefixes, Authorization headers,
  or token-like payloads.
- The generated startup hardening bullets target the existing identity startup
  subsystem. This task keeps them as regression anchors rather than broadening
  a PAL deployment manifest into startup rewiring.

## Acceptance Criteria

1. `pal-2026/ops/deployment-manifest.json` exists, loads through the typed
   manifest API, and declares `/opt/pal` as the deployment root.
   - **VERIFY:** `pytest tests/test_pal_deploy.py::test_default_pal_deployment_manifest_loads_from_repo -q`

2. The manifest lists the intended host-managed `/opt/pal` files, including
   router, startup scripts, shutdown config/script, deployment info, and Docker
   fallback files.
   - **VERIFY:** `pytest tests/test_pal_deploy.py::test_default_pal_deployment_manifest_lists_intended_opt_pal_files -q`

3. Every managed file target is inside `/opt/pal`, has a valid mode, has a
   unique target path, and required source files exist in the repo.
   - **VERIFY:** `pytest tests/test_pal_deploy.py::test_default_pal_deployment_manifest_has_valid_targets_modes_and_sources -q`

4. Runtime data paths are excluded from managed file entries and represented
   only as runtime/excluded path metadata.
   - **VERIFY:** `pytest tests/test_pal_deploy.py::test_default_pal_deployment_manifest_excludes_runtime_state_files -q`

5. The manifest and its typed payload contain no secrets or secret-like values.
   - **VERIFY:** `pytest tests/test_pal_deploy.py::test_default_pal_deployment_manifest_contains_no_secrets -q`

6. Invalid manifests fail clearly for duplicate targets, outside-root targets,
   malformed modes, and secret-looking content.
   - **VERIFY:** `pytest tests/test_pal_deploy.py::test_pal_deployment_manifest_validation_rejects_invalid_entries -q`

7. Existing PAL-focused regressions and startup identity hardening anchors stay
   green.
   - **VERIFY:** `pytest tests/test_pal_agent.py tests/test_pal_smoke.py tests/test_pal_client.py tests/test_pal_knowledge.py -q`
   - **VERIFY:** `pytest tests/test_cli_identity_hardening.py tests/test_first_boot.py tests/test_narrative_api_main.py -q`

8. Product-facing docs, changelog, and progress notes mention the PAL
   deployment manifest without claiming deploy apply or rollback support.
   - **VERIFY:** `rg -n "deployment manifest|deployment-manifest|pal_deploy" docs/architecture.md docs/command-reference.md pal-2026/docs/PROJECT_GUIDE.md CHANGELOG.md progress.md`

9. No new dependencies, migrations, remote-write commands, provider secrets, or
   database columns are introduced.
   - **VERIFY:** `git diff -- pyproject.toml promptclaw/coherence/migrations`
   - **VERIFY:** `pip install -e '.[dev]' && pytest tests/ -x && ruff check src/ tests/ && mypy src/`
