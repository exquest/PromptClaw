# Task T-011@20260502T001347Z: POST /beats/next

## Problem Statement

Deniable needs to request the next shared narrative `StoryBeat` from the
CypherClaw narrative engine over HTTP. The endpoint must wrap
`NarrativeEngine.next_beat()` rather than reimplement beat generation, accept
the Deniable cycle parameters `{cycle_number, domain_filter,
arc_position_target?, force_arc_event?}`, and return the serialized StoryBeat
JSON shape produced by the in-process engine call.

## Technical Approach

- Extend the existing `cypherclaw.narrative_api` package rather than adding a
  second service surface.
- Add a `NextBeatRequest` Pydantic schema:
  - `cycle_number`: non-negative integer.
  - `domain_filter`: required string, list of strings, or null, matching the
    existing `DomainFilter` alias used by memory/events.
  - `arc_position_target`: optional float in `[0.0, 1.0]`.
  - `force_arc_event`: optional non-empty string after stripping.
- Add a small `beats.py` helper module with:
  - `build_default_narrative_engine()` lazy-importing
    `cypherclaw.narrative.engine.NarrativeEngine`.
  - `generate_next_beat(...)` invoking the injected/default engine's
    `next_beat()` method.
  - `serialize_story_beat(...)` converting mappings, dataclasses, Pydantic
    models, and object-shaped beats into JSON-safe dictionaries.
- Mirror the established app factory pattern by accepting
  `narrative_engine` / `narrative_engine_factory` injection keywords in
  `create_app(...)`, and return HTTP `503` when the default deploy-side engine
  cannot be imported.
- Use signature inspection so the wrapper passes only parameters the
  deploy-side `next_beat()` accepts while supporting the expected keyword
  names. This keeps the HTTP wrapper thin if the in-process engine is still on
  a no-argument `next_beat()` implementation in older deployments.
- Preserve the engine's output shape. Do not attach a fixed response model that
  would add absent `StoryBeat` fields as `null` or empty lists; Deniable's
  bridge keeps the whole payload and only relies on `id`, `cycle_number`, and
  `arc_position`.
- Protect `POST /beats/next` with the same optional `X-Narrative-Auth` dependency
  used by the other non-health endpoints.
- Return HTTP `502` when the in-process engine raises during beat generation.
- Do not add migrations or dependencies; this endpoint does not create or alter
  database columns.

## Edge Cases

- Missing `domain_filter` is invalid because Deniable must explicitly ask for
  its domain view (`deniable` plus shared state is enforced by the engine/world
  layer, not inferred here).
- `arc_position_target` below `0.0` or above `1.0` is rejected before the engine
  is called.
- Whitespace-only `force_arc_event` is rejected before the engine is called.
- If `next_beat()` accepts `**kwargs`, pass all supported request fields.
- If `next_beat()` exposes only a subset of fields, pass that subset and still
  serialize the returned beat.
- If the engine returns a nested dataclass/object/Pydantic value inside the
  beat payload, recursively convert it to JSON-safe dictionaries/lists.
- If no deploy-side engine is importable and no test engine is injected, return
  HTTP `503` with a useful unavailable detail.
- The generated hardening items for SenseWeave self-critique,
  resource-governor enforcement, and daemon identity startup are unrelated to
  this endpoint. Existing startup/governor regression tests remain verification
  anchors only.
