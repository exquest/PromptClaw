# T-024@20260515T214233Z Spec: PAL Deploy-Plan CLI

## Problem Statement

PAL deployment tooling has a repo-managed manifest and a deterministic deploy
diff model, but operators do not yet have a CLI command that prints the dry-run
deployment plan. PAL-029 requires a command that prints the intended file plan
and service impact while performing no remote writes.

## Technical Approach

- Add `promptclaw pal deploy plan PROJECT_ROOT` under the existing PAL CLI group.
- Load `PROJECT_ROOT/ops/deployment-manifest.json` by default, with an optional
  `--manifest PATH` override for tests or diagnostics.
- Build a dry-run plan from `promptclaw.pal_deploy` by reusing the existing
  manifest loader, validation, fake/local remote inventory snapshots, and diff
  model.
- Support `--remote-inventory PATH` as a local JSON snapshot input only. This is
  not SSH and is not a live remote probe. When omitted, the command plans
  against an empty remote snapshot.
- Print a concise human-readable summary by default and a JSON-safe payload with
  `--json`.
- Expose explicit dry-run/no-write metadata in the payload and text output:
  `dry_run=true`, `remote_writes=false`, and no approval/apply flag.

## Edge Cases

- Missing or invalid manifests fail through the existing manifest validation path.
- Missing required local source files fail manifest validation before a plan is
  printed.
- Optional missing local source files are reported through the diff model as
  `missing`.
- Local inventory snapshots may be a target-keyed object or an object/list of
  remote-file records; malformed JSON or malformed entries fail before printing.
- Runtime paths excluded by the manifest remain ignored as unmanaged remote
  files.
- The command must not create `.promptclaw` run artifacts, SSH sessions, backups,
  service restarts, remote files, or approval-gated apply actions.

## Assumptions

- No new dependencies, migrations, provider secrets, database columns, live SSH
  reads, remote writes, backups, deploy apply, rollback, service restarts, or
  cloud lifecycle operations are required.
- The generated startup identity hardening bullets target the existing identity
  startup subsystem. Existing CLI, first-boot, daemon-ordering, and narrative
  ASGI tests remain mandatory regression anchors for this task.
