# T-026@20260515T214233Z Spec: Fake-Client CLI Tests for PAL Workflows

## Problem Statement

PAL now has local KB commands, approval-gated action execution, read-only
workflow CLIs, and deploy-plan output. Existing tests cover the underlying
helpers and many command functions directly, but PAL-041 asks for fake-client
CLI integration coverage that drives the public parser/dispatch path without
requiring a live PAL router, SSH connection, Tailscale state, or remote host.

## Technical Approach

- Add a focused fake-client CLI test module that calls `promptclaw.cli.main(...)`
  with real PAL command arguments while monkeypatching first-boot identity and
  PAL router client construction.
- Use local temp projects and `promptclaw.json` fixtures for every command.
- Cover the shipped approval surface as `promptclaw pal agent actions
  PROJECT_ROOT --approve ACTION_ID`. The PRD's artifact-replay command
  `promptclaw pal agent approve` is scheduled later as T-033, so this task does
  not add or test that future parser surface.
- Cover one read-only workflow command through the real CLI dispatch path:
  `promptclaw pal validate restart PROJECT_ROOT --json`, because T-026 depends
  on T-015 and restart validation exercises health, query, smoke, local command,
  and skipped SSH paths with a fake PAL client.
- Cover KB build/query and deploy-plan commands through `main(...)` while
  asserting they do not instantiate a PAL router client.
- Add a small structural test that pins the fake-client CLI coverage file and
  required test names.

## Edge Cases

- KB build/query must remain local-only and must not instantiate `PALRouterClient`.
- Deploy plan must remain local-only, dry-run, and avoid `.promptclaw` run
  artifacts.
- Approval testing must prove an approved allow-listed action executes from the
  fake model plan and that unknown approvals are not needed.
- Workflow testing must prove fake PAL health/query responses are enough for CLI
  dispatch and artifact creation while optional SSH/Tailscale evidence can be
  skipped without live infrastructure.
- Startup identity hardening remains a regression anchor. This task adds tests
  around PAL CLI coverage only and does not rewire startup.

## Assumptions

- No new dependencies, migrations, provider secrets, database columns, remote
  writes, approval replay command, deploy apply command, rollback command, or
  live PAL checks are required.
- "Approve" in this task maps to the currently available
  `pal agent actions --approve ACTION_ID` CLI path. The distinct
  `pal agent approve PROJECT_ROOT --run-id --action` parser is a later T-033
  requirement and remains out of scope.
