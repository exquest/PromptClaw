# T-011@20260515T214233Z Spec: Add Vast Connector Stub Boundary

## Problem Statement

PAL 2026 is deployed on a Vast.ai instance, but PromptClaw's PAL operator
workflow must not gain implicit authority to rent, destroy, start, or stop cloud
instances. The existing PAL action workflow executes only fixed action ids from
`build_default_action_registry(...)`, but there is no explicit Vast connector
boundary documenting that lifecycle operations are intentionally unavailable by
default.

## Technical Approach

- Add a small typed Vast connector boundary module under `promptclaw/`.
- Model Vast lifecycle action names as blocked metadata, not executable
  `PALOpsAction` entries.
- Expose a default stub object that reports:
  - provider: `vast`
  - blocked actions: `rent`, `destroy`, `start`, `stop`
  - callable actions: empty
  - status/reason explaining that the boundary is a non-executing stub
- Integrate the boundary into PAL action planning prompt context so PAL sees
  that Vast lifecycle operations are blocked and unavailable as action ids.
- Keep `build_default_action_registry(...)` unchanged for actual callable
  actions except for ensuring no Vast lifecycle ids are present.
- Continue to ignore unknown proposed action ids through `_parse_action_plan`.
- Do not add dependencies, secrets, migrations, remote write commands, HTTP
  routes, or a Vast API client.

## Edge Cases

- PAL proposes `rent`, `destroy`, `start`, or `stop`: those ids remain ignored
  because they are not registered actions.
- PAL proposes prefixed forms such as `vast_rent` or `vast_destroy`: those ids
  also remain ignored.
- A caller inspects the Vast stub directly: it returns no callable action names
  and has no lifecycle methods to invoke.
- Future work can replace or extend the boundary only by explicitly adding new
  callable actions and tests; this task does not create that execution path.
- Generated startup identity hardening bullets target existing startup code.
  This PAL/Vast boundary task keeps them as regression anchors rather than
  broadening scope into startup rewiring.

   `rent`, `destroy`, `start`, and `stop` as blocked.
   VERIFY: `pytest tests/test_vast_connector.py -q`

   and action-plan parsing ignores both bare and `vast_`-prefixed lifecycle
   proposals.
   VERIFY: `pytest tests/test_pal_agent.py::test_default_pal_action_registry_excludes_vast_lifecycle_actions -q`

3. PAL action-planning prompt artifacts include the Vast stub boundary and do
   not list Vast lifecycle operations as available fixed actions.
   VERIFY: `pytest tests/test_pal_agent.py::test_pal_ops_actions_prompt_includes_vast_stub_boundary -q`

4. Product documentation records that the Vast connector is a non-executing
   stub and that no `rent`, `destroy`, `start`, or `stop` action is callable by
   default.
   VERIFY: `grep -n "Vast connector" docs/architecture.md docs/command-reference.md docs/startup-wizard.md CHANGELOG.md progress.md`
