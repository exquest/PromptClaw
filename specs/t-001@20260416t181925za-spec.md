# Specification: T-001@20260416T181925Za

## Problem Statement

The CypherClaw humanization PRD needs a rule-ablation debugger so operators can
rerender a score with selected expression rules disabled and isolate which rule
caused an unwanted change. This split task builds only the core ablation engine:
given a score, seed context, a current active rule set, and rule IDs to disable,
the engine filters the disabled rules out, calls the existing render pipeline,
and returns that modified render output.

The broader render stack and concrete R1-R12 rule implementations do not exist
in this checkout yet. This task therefore defines the ablation seam as a small,
typed, stdlib-only utility that is generic over rule objects and renderer output.
Later render-pass tasks can plug their concrete `RenderPass` and rule classes
into the same function.

## Technical Approach

1. Add `my-claw/tools/senseweave/render/` as the future render package.
2. Add `senseweave.render.ablation.ablate()` with this contract:
   - accepts `score`, optional `seeds`, `disabled_rules`, `active_rules`, and an
     optional `renderer`;
   - extracts rule IDs from `.rule_id` first, then `.id`;
   - preserves the active rule order while filtering disabled rule IDs;
   - fails fast on unknown disabled rule IDs so listener-debug typos are visible;
   - calls the renderer with `score`, `seeds`, and the filtered `rules`;
   - returns the renderer output unchanged.
3. Provide a default no-rule renderer path that delegates to the existing
   `generative_scores.score_to_frequencies()` score rendering seam. Concrete
   active rules still require an explicit renderer until the render-pass
   framework lands.
4. Export the ablation API from `senseweave.render`.

## Edge Cases

- Empty disabled set: render with the active rule set unchanged.
- Single-rule ablation: only that rule is removed; rule order is otherwise
  unchanged.
- Pair ablation: multiple disabled IDs are deduplicated and removed together.
- All-rules-off: the renderer receives an empty rule tuple; output must match
  the renderer's baseline call exactly.
- Unknown disabled rule IDs: raise `ValueError` rather than silently returning a
  misleading ablation.
- Rule object without `.rule_id` or `.id`: raise `TypeError`.

## Assumptions

- No new dependencies are required.
- The concrete `RenderPass`, `PerformedPart`, and R1-R12 rule modules are owned
  by sibling/future tasks. This task keeps the core ablation function generic
  enough for those pieces to adopt without forcing their implementation here.
- The auto-generated startup identity hardening checks are unrelated to this
  pure render utility and are already covered by the existing startup tests.
  They remain explicit verification anchors for this task.

## Acceptance Criteria

1. Single-rule ablation filters exactly the requested rule and re-renders through
   the supplied renderer with seeds preserved.
   VERIFY: `pytest tests/test_render_ablation.py::TestAblate::test_single_rule_ablation_filters_rule_and_rerenders -q`

2. Pair ablation removes both requested rules while preserving the relative order
   of the remaining active rules.
   VERIFY: `pytest tests/test_render_ablation.py::TestAblate::test_pair_ablation_preserves_remaining_rule_order -q`

3. All-rules-off ablation passes an empty rule tuple to the renderer and matches
   the renderer's baseline output exactly.
   VERIFY: `pytest tests/test_render_ablation.py::TestAblate::test_all_rules_off_matches_baseline_render -q`

4. Unknown disabled rule IDs are rejected as configuration errors.
   VERIFY: `pytest tests/test_render_ablation.py::TestAblate::test_unknown_disabled_rule_id_raises -q`

5. Startup identity persistence remains covered for standalone and federated
   startup paths.
   VERIFY: `pytest tests/test_first_boot.py::TestStartupIdentityPersistence tests/test_governor_integration.py::TestStartupIdentityWiring -q`
