# Specification: T-013@20260425T183959Za

## Problem Statement

`docs/cypherclaw-musicianship-roadmap.md` still describes CypherClaw through
the older quartet-era songwriting lens. Since that document landed, the sampler
PRD and follow-on tasks introduced a new artistic direction:

- the ensemble is intended to expand from a signature quartet to a quintet
- the sampler is not just another synth voice, but a memory-bearing role for
  room capture, self-quotation, and curated found sound
- sampler implementation status is uneven across the tree, so the roadmap now
  needs to distinguish what is already present from what remains on the roadmap

Today the repository has a mixed state. Sampler capture, dispatch, effects-bus,
and anti-pattern work exist, but the canonical identity module and its tests
still describe a quartet. The roadmap should reflect that reality instead of
claiming the quintet is already fully wired end to end.

## Technical Approach

1. Update `docs/cypherclaw-musicianship-roadmap.md` with a new ensemble
   snapshot near the top of the document that:
   - names the five intended signature voices:
     `sw_bell_warm`, `sw_bowed`, `sw_breath`, `sw_pad`, `sw_sampler`
   - describes their current musical-role model:
     melody/clear-tone lead, bowed counterline, breath/noise color, sustained
     harmonic foundation, and sampler memory voice
   - explains that the sampler role expands the artistic target from quartet to
     quintet
2. Add a concise implementation-status section that separates:
   - landed sampler pieces already present in the tree
   - still-pending pieces required before the quintet is fully authoritative in
     runtime identity/cast selection
3. Keep the rest of the roadmap intact, but update any stale quartet-only
   language that would contradict the new ensemble snapshot and status section.
4. Add a focused regression test for the roadmap document so future edits
   preserve the quintet composition, role mapping, and partial-rollout status.

## Edge Cases

- The document must not overstate completion. The sampler exists in parts of
  the stack, but `artist_identity.py` and `tests/test_artist_identity.py` still
  encode a quartet, so the roadmap needs to call out that mismatch explicitly.
- The document should not pretend the sampler is a generic texture voice; it
  must be described as a memory-bearing instrument tied to room capture,
  self-quotation, and found-sound library work.
- The status snapshot should reference concrete modules so a reader can verify
  what has landed versus what remains planned.
- The update should stay scoped to the roadmap doc plus task-tracking artifacts;
  it must not silently change runtime code to force the docs to become true.

## Assumptions

- For this subtask, “reflect the quintet” means documenting the intended
  five-voice ensemble and the current partial landing status, not falsely
  claiming that every runtime entry point already uses the quintet.
- No new dependencies or migrations are required.
- The remaining `CCS-033` documentation work (`sampler-architecture` and
  `sampler-artistic-intent` docs) belongs to sibling subtasks, not this one.

## Acceptance Criteria

1. The roadmap names the intended signature quintet and the sampler’s
   memory-bearing role.
   VERIFY: `pytest tests/test_cypherclaw_musicianship_roadmap.py::test_roadmap_names_signature_quintet_and_sampler_role -q`

2. The roadmap records the current partial-rollout status by distinguishing
   landed sampler modules from still-pending quintet-integration work.
   VERIFY: `pytest tests/test_cypherclaw_musicianship_roadmap.py::test_roadmap_records_landed_and_pending_quintet_status -q`

3. The focused roadmap regression test passes end to end.
   VERIFY: `pytest tests/test_cypherclaw_musicianship_roadmap.py -q`
