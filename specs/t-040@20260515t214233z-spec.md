# T-040@20260515T214233Z Spec: PAL Deploy Backup Primitive

## Problem Statement

PAL deployment tooling can load the repo-managed manifest, compare it against a
fake or saved remote inventory, and print a dry-run plan. Before any future
approved deploy-apply command can overwrite remote managed files, PromptClaw
needs a typed backup primitive that stores the existing remote versions of
changed files in a local deployment artifact.

This task implements PAL-030 from
`sdp/prd-pal-2026-agentic-ops-platform.md`: fake remote test stores changed
files. It must stay local and deterministic for tests and future apply/rollback
work.

## Technical Approach

- Extend `promptclaw.pal_deploy` with a backup model that records:
  backup id, manifest path, deployment root, local backup root, stored entries,
  and explicit remote-write status.
- Preserve fake remote content inside `PALDeploymentRemoteFile` instances built
  from `build_fake_pal_remote_inventory(...)` so backup code can materialize the
  previous remote bytes without live SSH.
- Add `backup_pal_deployment_changes(...)` that:
  - computes the existing manifest diff against a remote inventory,
  - selects only `changed` managed files,
  - writes each changed remote file under
    `<backup_root>/<backup_id>/files/<target-without-leading-slash>`,
  - writes a JSON-safe `backup-manifest.json`, and
  - returns a typed backup result for later apply/rollback work.
- Keep `added`, `missing`, `unchanged`, and unmanaged remote files out of the
  backup payload because they are not existing managed files that would be
  overwritten by apply.
- Keep the primitive stdlib-only and independent of CLI parsing, SSH, remote
  writes, service restarts, deploy apply, and rollback.

## Edge Cases

- Output order must be deterministic by target path.
- Metadata-only changed files must be backed up because rollback may need their
  previous mode, owner, or group metadata even when content hashes match.
- Fake remote inventory entries without content are not backupable; the
  primitive should fail clearly rather than write an empty or misleading backup.
- Backup paths must be derived from normalized remote targets and must not allow
  parent traversal.
- The primitive writes local artifact files only; it must not mutate the remote
  inventory or add CLI approval/apply behavior.
- Startup identity hardening remains an existing regression anchor. This PAL
  backup task does not change startup flow because current CLI, daemon,
  first-boot, and narrative ASGI tests already cover `bootstrap_identity()`
  persistence and ordering before `FirstBootAnnouncer` in standalone and
  federated modes.

- The backup primitive is a code API and local artifact writer only; T-041 owns
  approved deploy-apply CLI behavior, and T-042/T-043 own rollback behavior.

  SSH backup capture remains future work behind the apply/deploy approval gate.
