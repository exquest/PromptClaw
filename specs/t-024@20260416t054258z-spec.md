# Task T-024 Specification: Executable Production-Course Curriculum Exercises

## Problem Statement

The EMSD curriculum already has a 40-course catalog, generated course scaffolds,
and a small verifier-stub layer. The production-course runtime also already
defines the musical concepts that every generated piece carries through score
tree and tracker metadata:

- mode/scale
- harmonic function and progressions
- meter/groove
- counterpoint relation
- synthesis architecture
- mix role and spatial intent
- genre strategy
- phase profile and transitions

The gap is that the curriculum does not yet expose these production-course
concepts as executable exercises. Existing exercise metadata has an objective,
verifier, and expected keys, but it lacks a reusable starter template and uses
`expected` rather than the more explicit `expected_features` name. Several
relevant courses also still have no exercises, so the curriculum does not yet
cover intervals/scales, chord voicings, progressions, rhythm/groove,
counterpoint, SenseWeave mapping, Theramini ensemble behavior, genre strategy,
and full-arc production as runnable curriculum work.

## Technical Approach

Extend the existing `my-claw/curriculum` pattern instead of adding a parallel
exercise system.

1. Update `ExerciseSpec` in `my-claw/curriculum/catalog.py` with:
   - `objective`
   - `template`
   - `verifier`
   - `expected_features`
   - a backward-compatible `expected` property for existing callers/tests.

2. Update `_ex()` to accept a template payload and expected feature tuple.

3. Add production-course exercise coverage across the catalog:
   - intervals/scales: `EMSD-110`
   - chord voicings: `EMSD-210`
   - progressions: `EMSD-302`
   - rhythm/groove: `EMSD-220`
   - counterpoint: `EMSD-302`
   - synthesis architecture: `EMSD-201`
   - mix/master: `EMSD-301`
   - SenseWeave mapping: `EMSD-257`
   - Theramini ensemble: `EMSD-251`
   - genre strategy: `EMSD-253`
   - full arc production: `EMSD-499`

4. Keep verifier stubs lightweight and deterministic:
   - `structural`: required keys present
   - `constraint`: required keys present and not false/none
   - `spectral`: constraint alias
   - `temporal`: constraint alias

5. Update `materialize_curriculum()` so generated `spec.json` files include
   both `expected_features` and legacy `expected`, and exercise READMEs show a
   starter template.

6. Regenerate the committed curriculum scaffolds from the catalog so the source
   catalog and product-facing curriculum tree stay consistent.

## Edge Cases

- Existing tests or callers using `ex.expected` must continue to work.
- Existing `spec.json` consumers using the legacy `expected` key must continue
  to work.
- Exercises whose templates contain nested data should still pass the current
  verifier stubs when all expected feature keys are present and truthy.
- Courses with more than one exercise must produce unique exercise IDs.
- All verifier kinds declared by exercises must be callable through
  `verify_exercise()`.
- Materializing into a temporary directory must create every exercise directory
  and spec file listed by `scaffold_relpaths()`.

## Assumptions

- The runtime startup hardening items generated for this task are not part of
  the curriculum feature surface. They are already addressed in the current
  tree: both daemon entrypoints call `bootstrap_identity()` before
  `FirstBootAnnouncer`, and `tests/test_first_boot.py` plus
  `tests/test_governor_integration.py` cover identity persistence and wiring.
- No new third-party dependency is required.
- Database migrations are not applicable because the curriculum catalog and
  generated scaffolds are static files and pure Python helpers.

## Acceptance Criteria

1. Curriculum includes executable exercises for intervals/scales, chord
   voicings, progressions, rhythm/groove, counterpoint, synthesis architecture,
   mix/master, SenseWeave mapping, Theramini ensemble, genre strategy, and full
   arc production.
   VERIFY: `pytest tests/test_executable_curriculum.py::test_production_course_exercise_topics_are_covered -q`

2. Exercise metadata includes objective, template, verifier, and expected
   features in both catalog objects and generated `spec.json` files.
   VERIFY: `pytest tests/test_executable_curriculum.py::test_exercise_metadata_includes_templates_and_expected_features -q`

3. Tests verify curriculum completeness and runnable verifier stubs.
   VERIFY: `pytest tests/test_executable_curriculum.py::test_every_exercise_template_passes_its_verifier_stub -q`

4. Existing EMSD catalog and scaffold behavior remains compatible.
   VERIFY: `pytest tests/test_emsd_curriculum.py -q`

5. Startup identity hardening remains covered without changing unrelated
   startup code.
   VERIFY: `pytest tests/test_first_boot.py tests/test_governor_integration.py -q`
