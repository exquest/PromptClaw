# Spec: T-018@20260425T183959Zd

## Problem Statement

`T-018@20260425T183959Za` taught the MIDI keyboard and Theramini listeners to
emit `phrase_started` / `phrase_ended` after >5 seconds of continuous play.
`T-018@20260425T183959Zb` added `PhraseCaptureWriter`, and
`T-018@20260425T183959Zc` added validated sidecar tags. The remaining gap is
the one called out by the Zc verifier: the production listeners still never
call the writer, so no active-song human phrase actually lands in
`samples/keyboard/` or `samples/theramini/`.

This split closes that gap by wiring active-song phrase capture into the live
listeners, deriving tags from the existing composer authority file, and adding
a deterministic verification path that simulates a >5 second keyboard and
Theramini phrase without requiring hardware.

## Technical Approach

### 1. Add a small active-song capture runtime wrapper

Create a SenseWeave helper module that wraps `PhraseCaptureWriter` with the
runtime concerns the listeners need:

- Read `/tmp/composer_state.json` safely
- Treat it as active-song context only when it contains a current song id
- Map composer state into the locked sidecar schema:
  - `instrument`: `"midi_keyboard"` for keyboard captures, `"theramini"` for
    Theramini captures
  - `song_id`: `str(composer_state["song"])` when present
  - `key`: `composer_state["key"]` when present
  - `tempo`: one of `tempo`, `tempo_bpm`, or `bpm` when present and numeric
  - `source`: `"human"`
- Reset buffered phrase state when no active song is available or the song id
  changes mid-phrase

The wrapper should expose a simple `feed(chunk, is_playing, now)` method so the
listeners do not need to know about sidecar metadata internals.

### 2. Keep `PhraseCaptureWriter` generic, but make reset explicit

Add a public reset path on `PhraseCaptureWriter` so the runtime wrapper can
discard a partial phrase cleanly when:

- there is no active song
- the active song changes
- the caller wants to abandon buffered state after a capture decision boundary

No schema changes are allowed in this split. The sidecar must remain limited to
`instrument`, `song_id`, `key`, `tempo`, `timestamp`, `duration`, and
`source`.

### 3. Wire the listeners into the wrapper

Update both listeners to use the wrapper:

- `midi_keyboard_listener.py`
  - Adapt listener state into `capture.feed(...)` using the serialized
    `playing` / `timestamp` fields
  - Feed raw MIDI bytes (or `None` on no-event ticks) so note-on / note-off
    timing still lands in the SMF
- `theramini_listener.py`
  - Feed the final post-idle-tone `is_playing` decision, not the pre-override
    analysis result
  - Feed the decoded mono audio chunk from the captured WAV so the written WAV
    reflects the phrase the listener actually heard

Small listener-local helper functions are acceptable if they keep the runtime
tests focused and avoid duplicating state-field mapping logic.

### 4. Add a deterministic verification entry point

Add a hardware-free verifier script that:

- writes a synthetic active-song `composer_state.json`
- simulates a >5 second keyboard phrase and a >5 second Theramini phrase
- asserts that one file appears under `samples/keyboard/` and one under
  `samples/theramini/`
- validates both sidecars with `validate_sample_metadata`
- prints a success marker for operator use

This verifier is the manual verification surface for the task, but it must be
fully covered by automated tests.

## Edge Cases

- No composer state file: ignore input and write nothing
- Composer state present but missing `song`: ignore input and write nothing
- Composer state missing tempo: allow `tempo = None`
- Song id changes before phrase end: discard buffered phrase and start fresh for
  the new song
- Keyboard note-off / Theramini release tick must still be included in the
  qualifying phrase boundary
- Sub-threshold bursts during an active song still write nothing
- Idle-tone-suppressed Theramini ticks must not count as human phrase capture

## Acceptance Criteria

### AC-1: Active-song metadata is derived from `composer_state.json`
The runtime wrapper must convert the current composer state into the locked
sidecar schema and reject inactive or song-less state.

**VERIFY:** `pytest tests/test_phrase_capture_runtime.py::test_read_active_song_metadata_maps_song_key_and_tempo -v`

### AC-2: Keyboard listener integration writes a qualifying phrase into `samples/keyboard/`
Simulating a >5 second MIDI phrase during an active song must produce a `.mid`
file under `samples/keyboard/` plus a valid sidecar carrying the active song's
`song_id`, `key`, and `tempo`.

**VERIFY:** `pytest tests/test_midi_keyboard_listener_runtime.py::test_capture_phrase_writes_keyboard_phrase_during_active_song -v`

### AC-3: Theramini listener integration writes a qualifying phrase into `samples/theramini/`
Simulating a >5 second Theramini phrase during an active song must produce a
`.wav` file under `samples/theramini/` plus a valid sidecar carrying the active
song's `song_id`, `key`, and `tempo`.

**VERIFY:** `pytest tests/test_theramini_listener_runtime.py::TestAnalyzeOnce::test_capture_phrase_writes_theramini_phrase_during_active_song -v`

### AC-4: Inactive-song input is ignored and buffered state is dropped
When there is no active song context, listener input must not produce capture
files, and stale partial phrases must not leak into the next active song.

**VERIFY:** `pytest tests/test_phrase_capture_runtime.py::test_inactive_song_resets_partial_phrase_and_writes_nothing -v`

### AC-5: Instrument tags use listener-specific values while staying schema-valid
Keyboard captures must tag `instrument = "midi_keyboard"` and Theramini
captures must tag `instrument = "theramini"` without changing the locked sidecar
key set.

**VERIFY:** `pytest tests/test_phrase_capture_runtime.py::test_runtime_overrides_instrument_tag_per_source -v`

### AC-6: Manual verification script succeeds without hardware
The verifier script must simulate both human-input sources against an active
song, print a success marker, and leave the expected files in the correct
directories.

**VERIFY:** `pytest tests/test_human_phrase_capture_verify.py::test_main_prints_human_phrase_capture_ok_and_returns_zero -v`
