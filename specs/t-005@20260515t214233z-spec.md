# T-005@20260515T214233Z Specification

## Problem Statement

PAL-014 extends the local PAL knowledge-base slice after PAL-013 source
discovery. Future PAL index-writing and prompt-injection tasks need a stable,
inspectable chunk stream built from configured PAL source files. The chunker
must be deterministic: unchanged input files, configuration, and chunking
options must produce the same chunk ids across repeated runs.

The acceptance criterion is: "Test proves stable chunk ids for unchanged input
files."

## Technical Approach

- Keep the implementation in `promptclaw/pal_knowledge.py` beside
  `discover_pal_source_files(...)`.
- Add a frozen `PALKnowledgeChunk` dataclass with typed fields:
  `chunk_id`, `source_path`, `source_sha256`, `chunk_index`, `start_line`,
  `end_line`, and `text`.
- Add `chunk_pal_source_files(project_root, config=None, max_chars=2000)`:
  - uses `discover_pal_source_files(...)` for deterministic source order;
  - reads each file locally as UTF-8 text;
  - normalizes line endings through `splitlines()`;
  - packs complete lines into chunks up to `max_chars` where possible;
  - splits a single overlarge line only when required;
  - skips empty/whitespace-only files;
  - returns chunks ordered by source order, then chunk index.
- Compute `source_path` as a POSIX path relative to `project_root` when the
  source is inside the project; otherwise use the resolved absolute POSIX path.
- Compute `source_sha256` from the normalized file text.
- Compute `chunk_id` from stable fields only: source path, source hash, chunk
  index, line range, and chunk text. Do not include runtime state, wall-clock
  time, absolute temp paths for in-project files, or discovery iteration order.
- Keep the implementation stdlib-only and side-effect free. This task does not
  write `.promptclaw/pal-kb/`; PAL-015 owns the index writer.

## Edge Cases

- Repeated chunking of unchanged files must produce identical chunk ids and
  identical chunk payloads.
- Adding an unrelated source file must not change ids for existing unchanged
  files.
- Overlapping source patterns can discover a file once; chunking inherits the
  source discovery de-duplication behavior.
- Empty or whitespace-only files yield no chunks.
- A blank line inside a non-empty file remains part of chunk text and line
  numbering.
- A single line longer than `max_chars` is split deterministically into
  bounded text chunks with the same source line number.
- Invalid `max_chars` values raise `ValueError` before reading files.
- This task introduces no database columns, migrations, provider secrets, or
  external dependencies.
- The generated startup identity hardening bullets target the existing
  identity/startup subsystem. Current startup tests already cover
  `bootstrap_identity()` persistence and ordering before `FirstBootAnnouncer`;
  those anchors must be re-run, but PAL chunking does not modify startup flow.

## Acceptance Criteria

1. PAL source chunking returns typed chunks with deterministic metadata,
   bounded text, source-relative paths, and JSON-safe payloads.
   VERIFY: `pytest tests/test_pal_knowledge.py::test_chunk_pal_source_files_returns_typed_bounded_chunks -q`

2. Repeated chunking of unchanged input files produces stable chunk ids and
   stable payloads.
   VERIFY: `pytest tests/test_pal_knowledge.py::test_chunk_pal_source_files_keeps_chunk_ids_stable_for_unchanged_files -q`

3. Adding an unrelated configured source file does not change chunk ids for
   already-unchanged input files.
   VERIFY: `pytest tests/test_pal_knowledge.py::test_chunk_pal_source_files_keeps_existing_ids_when_new_source_is_added -q`

4. Edge cases are deterministic: whitespace-only files are ignored, overlong
   lines split without exceeding `max_chars`, and invalid `max_chars` raises
   `ValueError`.
   VERIFY: `pytest tests/test_pal_knowledge.py::test_chunk_pal_source_files_handles_empty_and_long_line_edges tests/test_pal_knowledge.py::test_chunk_pal_source_files_rejects_invalid_max_chars -q`
