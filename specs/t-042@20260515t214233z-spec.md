# T-042@20260515T214233Z Spec: PAL Deploy Rollback Primitive

## Problem Statement

PAL deployment tooling can produce dry-run diffs, store changed managed fake
remote files in local backup artifacts, and apply approved changes to a local
fake remote inventory snapshot. PromptClaw still needs the rollback primitive
that restores the backed-up files into that fake remote inventory so a later CLI
can expose rollback behind an explicit approval gate.

This task implements PAL-032 from
`sdp/prd-pal-2026-agentic-ops-platform.md`: fake remote test restores backed-up
files. It stays local, deterministic, and API-level; T-043 owns
`promptclaw pal deploy rollback --approve-rollback`.

## Technical Approach

- Extend `promptclaw.pal_deploy` with typed rollback result models that record:
  status, approval status, manifest path, deployment root, remote inventory
  path, backup metadata, restored entries, summary counts, and explicit
  `live_ssh=false` / `service_restarts=false`.
- Add a primitive such as `rollback_pal_deployment_backup(...)` that:
  - requires an explicit `approved=True` argument before mutating the local fake
    remote inventory snapshot,
  - accepts a `PALDeploymentBackup` returned by the backup primitive,
  - reads each backed-up file from `backup.backup_path`,
  - rewrites only the backed-up targets in the supplied remote inventory,
  - restores backed-up content plus mode, owner, group, size, sha256, and source
    metadata,
  - preserves unmanaged, excluded, added, unchanged, and non-backed-up current
    remote inventory entries, and
  - writes the resulting fake remote inventory JSON with the existing snapshot
    writer.
- Keep the implementation stdlib-only and independent of CLI parsing, live SSH,
  service restarts, deployment metadata, and provider secrets.

## Edge Cases

- Rollback without explicit approval must fail before writing the fake remote
  inventory snapshot.
- Missing backup files must fail clearly rather than writing empty content.
- Metadata-only backups must restore prior mode, owner, and group even when
  content is unchanged.
- Non-UTF-8 backup content must round-trip through JSON safely via the existing
  `content_base64` inventory payload support.
- Restoring a backed-up file should recreate it in the fake inventory even if a
  later manual edit removed that target from the current inventory.
- Runtime excluded and unmanaged files that were not in the backup artifact must
  stay untouched.
- Startup identity hardening remains an existing regression anchor. This PAL
  rollback task does not change startup flow because current CLI, first-boot,
  daemon-ordering, and narrative ASGI tests already cover `bootstrap_identity()`
  persistence and ordering before `FirstBootAnnouncer` in standalone and
  federated modes.

## Assumptions

- The rollback primitive is a code API and local fake-remote inventory writer
  only. T-043 owns CLI parser/dispatch and the `--approve-rollback` flag.
- The backup artifact from T-040 is authoritative for restore content and
  previous metadata.
- Live remote rollback capture/write support remains future work and must be
  separately approval-gated.
