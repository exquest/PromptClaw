# Task T-032c: Verify steady-state daily cost under cap

## Problem Statement

T-032a documented the SDP telemetry cost model, and T-032b produced
`sdp/telemetry/steady-state-cost.csv` with seven active UTC days of rolled-up
SDP token spend. The remaining CCG-032 requirement is to verify each of those
days against the daily USD cap, flag breaches, and publish a short PASS/FAIL
summary that links back to the cost model document.

No checked-in `/tmp/generation_status.json` soak snapshots or daily generation
spend chart are present in this repository. This subtask therefore verifies the
committed T-032b steady-state SDP cost CSV. Because the cost model document's
only concrete daily USD cap is the default `$5.00` daily cap, the verifier uses
`$5.00` by default while allowing callers to override the cap.

## Technical Approach

- Extend `promptclaw/sdp_cost.py` with a small cap-verification layer:
  - a default daily cap constant set to `$5.00`;
  - a typed `DailyCapCheck` result per day;
  - a typed `CostCapSummary` carrying `PASS` / `FAIL`, checks, breaches, and
    minimum-day validation;
  - `check_daily_caps(...)`, `summarize_daily_cap(...)`,
    `render_cap_summary(...)`, and `write_cap_summary(...)` helpers.
- Add `read_csv(...)` so the committed steady-state CSV can be re-read and
  verified without touching runtime state.
- Extend the `python -m promptclaw.sdp_cost` CLI with optional
  `--daily-cap-usd`, `--min-days`, and `--summary-out` arguments. Existing CSV
  extraction behavior remains unchanged when `--summary-out` is omitted.
- Generate and commit `sdp/telemetry/steady-state-cap-summary.md` from the live
  `.sdp/state.db` / CSV data. The summary must link to
  `sdp/telemetry/cost-model.md`, list every checked day, and mark any breach.
- Treat generated hardening items about self-critique wiring, resource-governor
  enforcement, and startup identity sequencing as verification anchors outside
  this task's implementation scope; the affected files here do not participate
  in composition/playback, resource governance, or daemon startup.

## Edge Cases

- Exactly-at-cap costs pass because the cost model rejects only when spend would
  become greater than the cap.
- Costs above cap are breaches and show the overage amount.
- Fewer than seven rows produce a `FAIL` summary even if all observed rows are
  under cap, because CCG-032 requires at least seven days.
- Empty CSVs, malformed CSVs, or missing required columns should fail loudly
  instead of silently publishing a misleading pass.
- Calendar gaps remain visible; per T-032b, this repository's steady-state CSV
  uses the seven most recent active UTC days rather than seven consecutive
  calendar dates.

## Acceptance Criteria

1. Cap verification functions return one typed check per daily cost row, treat
   cost `<= cap` as pass, and flag cost `> cap` as a breach with an overage.
   - **VERIFY:** `pytest tests/test_sdp_cost_extractor.py::test_check_daily_caps_flags_breaches_and_exact_cap_passes -q`

2. Summary rendering reports `PASS` only when at least seven days are present
   and none breach the cap; otherwise it reports `FAIL`, includes breach count,
   and links to `sdp/telemetry/cost-model.md`.
   - **VERIFY:** `pytest tests/test_sdp_cost_extractor.py::test_render_cap_summary_links_cost_model_and_lists_breaches tests/test_sdp_cost_extractor.py::test_cap_summary_fails_when_minimum_days_missing -q`

3. The CLI can write a cap summary while preserving the existing CSV extraction
   workflow; it exits non-zero only when the summary fails.
   - **VERIFY:** `pytest tests/test_sdp_cost_extractor.py::test_cli_writes_cap_summary_and_returns_failure_on_breach -q`

4. `sdp/telemetry/steady-state-cap-summary.md` is committed, links to
   `cost-model.md`, checks seven daily rows, and flags every breach visible in
   the current `sdp/telemetry/steady-state-cost.csv`.
   - **VERIFY:** `python -c "from pathlib import Path; text=Path('sdp/telemetry/steady-state-cap-summary.md').read_text(); assert '[sdp/telemetry/cost-model.md](cost-model.md)' in text and text.count('| 2026-') == 7 and 'BREACH' in text"`

5. Product-facing change notes and progress tracking mention the cap summary.
   - **VERIFY:** `grep -n "steady-state-cap-summary.md" CHANGELOG.md progress.md`
