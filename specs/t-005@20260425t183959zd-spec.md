# Specification: T-005@20260425T183959Zd

## Problem Statement

Subtasks Za–Zc added the `sampler_amp` (peak-normalized sampler gain) and
`sampler_bypass_comp` (post-mix compression bypass) controls to
`master_bus.py` and proved both controls flow through
`master_bus_values_for_scene`, `master_bus_n_set_args`, and
`master_bus_s_new_args` to OSC. The Python helper now sends them on every
scene change.

What's still missing is the *server-side* end of the contract:

1. There is no source-of-truth `master_smooth.scd` stub in the repo. The
   `sw_master_smooth.scsyndef` binary is the only artifact, so a future
   contributor reading the codebase has no way to confirm that
   `sw_master_smooth` actually declares `sampler_amp` and
   `sampler_bypass_comp` as control inputs. The runtime would silently drop
   them if the SynthDef did not.
2. The two boot scripts that seed the master node — `start_audio.sh` and
   `restart_composer.sh` — bypass `master_bus_s_new_args` and use a
   hardcoded `/s_new sw_master_smooth` arg list that omits both new
   controls. After a fresh boot or pop-safe restart, the master node would
   start with whatever defaults the SynthDef carries until the first scene
   change overwrites them. If the SynthDef defaults drift (for example, if
   the binary is rebuilt with `sampler_amp = 0`), the sampler bus would be
   silent on boot.

This subtask closes both gaps by introducing the SuperCollider stub and by
routing the boot-time `/s_new` through the same Python helper that scene
changes already use.

## Technical Approach

1. Add a SuperCollider source stub at
   `my-claw/tools/senseweave/synthesis/master_smooth.scd` that declares the
   `sw_master_smooth` SynthDef with its full control surface — including
   the new `sampler_amp` (default 1.0) and `sampler_bypass_comp`
   (default 1.0) — and writes a recompiled `.scsyndef` into
   `synthdefs/`. This mirrors the existing `sampler_effects.scd` pattern.
   The stub documents the chain the Python helper drives so the binary
   artifact and the OSC contract stay in sync.
2. Refactor the inline Python heredocs in `start_audio.sh` and
   `restart_composer.sh` so they import
   `senseweave.master_bus.master_bus_s_new_args` and pass its return list
   directly to `client.send_message("/s_new", ...)`. This makes
   `master_bus_s_new_args` the single source of truth for the seeded
   control surface and prevents future drift.
3. Run the full test suite to confirm no regression, plus `ruff` and
   `mypy` per the standard validation checklist.

No changes to `sampler_dispatch.py` itself are required — its `/s_new
sw_sampler` bundle does not carry master-bus controls. The "callers /
automation lanes referenced by sampler_dispatch.py" in the task are the
boot scripts that seed the *master* node the sampler ultimately routes
through.

## Edge Cases

- The `.scd` stub is a textual artifact only; SuperCollider is not
  available in CI, so the binary `.scsyndef` continues to be the compiled
  artifact. Tests assert the stub declares the new control names and
  defaults but do not attempt to compile or execute SuperCollider code.
- Boot scripts must remain idempotent: re-running them after a partial
  boot must still produce the same seeded master node. The refactor keeps
  the pre-`/s_new` `client.send_message("/g_freeAll", [0])` and post-spawn
  re-seed behaviour intact in `restart_composer.sh`.
- Tests verify the bash scripts via plain string-content checks (read the
  file, assert key tokens are present). They do not exec the scripts.
- The Python heredoc in the bash scripts must work under
  `/home/user/cypherclaw/.venv/bin/python3` with `PYTHONPATH` already set
  to `/home/user/cypherclaw/src:/home/user/cypherclaw/tools` (as
  `restart_composer.sh` already does). `start_audio.sh` does not currently
  set `PYTHONPATH`, so the refactor adds it inline before the heredoc.

## Assumptions

- Documenting the SynthDef as a `.scd` stub is sufficient; recompiling
  the binary `.scsyndef` is an out-of-band operator step (run the stub
  in a SuperCollider IDE), tracked elsewhere if it becomes necessary.
- Boot-time defaults come from `DEFAULT_MASTER_BUS_VALUES`; tests pin the
  defaults that ship from `master_bus_s_new_args()` rather than the raw
  bash text.
- The candidate hardening items about `bootstrap_identity()` and self-
  critique pipeline integration are out of scope: this subtask only
  closes the SynthDef / boot-script gap for the new master-bus controls.
- No new third-party dependencies, no migrations.
