# Specification: T-001@20260425T183959Zd

## Problem Statement

`BufferLoader` already supports `on_sampler_load()`, `on_sampler_free()`, and
basic LRU bookkeeping, but the capacity-overflow path is still implemented by a
private `_evict_lru()` helper that duplicates the release logic instead of
reusing the public free path. That leaves the cap enforcement logic slightly
out of sync with the contract implied by the task graph: when the 64-buffer cap
is reached, the least-recently-used live record should be evicted through the
same release path used by explicit frees before a new buffer is allocated.

The current tests also only exercise small-capacity eviction in unit-style
scenarios. There is no regression test that fills the default-capacity pool and
proves the full load/evict cycle works end to end.

## Technical Approach

Keep the existing `BufferLoader` interface and OSC behavior, but tighten the
eviction path.

1. Add fail-fast capacity validation in `BufferLoader.__init__()` so invalid
   `capacity < 1` configurations raise `ValueError`, matching the nearby
   `MotifMemory` fixed-capacity pattern.
2. Replace the private overflow-only eviction shortcut with a public
   `evict_lru()` helper that:
   - finds the oldest tracked record
   - delegates the actual release and bookkeeping to `on_sampler_free(record)`
   - returns the evicted record (or `None` when nothing is tracked)
3. Update `on_sampler_load()` so when `len(self._lru) >= self.capacity`, it
   calls `evict_lru()` before allocating the new buffer.
4. Add a regression test that:
   - fills the loader to the default 64-buffer capacity
   - triggers a 65th load
   - asserts the oldest record is evicted
   - asserts eviction happens through `on_sampler_free()`
   - asserts `/b_free` is sent before the new `/b_alloc` and `/b_allocRead`

## Edge Cases

- `capacity=0` or any negative capacity should raise `ValueError` at
  construction time instead of failing later with `KeyError`.
- `evict_lru()` should return `None` when no live records are tracked.
- Explicit free behavior must remain idempotent and continue removing the freed
  record from LRU accounting.
- Touch-based LRU promotion must keep working after the eviction path is
  rerouted through `on_sampler_free()`.

## Assumptions

- This split task continues the existing `BufferLoader` implementation rather
  than renaming it to `BufferPool`; the public behavior is the priority in this
  slice.
- The "integration test" requirement is satisfied by a filesystem-backed,
  multi-record pytest that exercises the real loader code path with mocked OSC
  sends rather than a live scsynth process.
- The auto-generated startup hardening checks are out of scope for this task
  because sampler buffer management does not participate in daemon startup or
  identity bootstrapping.
- No new third-party dependencies or database migrations are required.

## Acceptance Criteria

1. When `on_sampler_load()` is called with the default 64-buffer cap already
   full, the least-recently-used record is evicted before allocating the new
   buffer, and the eviction is routed through `on_sampler_free()`.
   VERIFY: `pytest tests/test_sampler_buffers.py::test_on_sampler_load_evicts_lru_via_on_sampler_free_at_default_capacity -q`

2. LRU promotion semantics remain intact after the eviction path is rerouted.
   VERIFY: `pytest tests/test_sampler_buffers.py::test_touch_promotes_record_to_most_recently_used -q`

3. Invalid non-positive capacities fail fast with `ValueError`.
   VERIFY: `pytest tests/test_sampler_buffers.py::test_buffer_loader_requires_positive_capacity -q`

4. The sampler buffer suite passes end to end without regressions.
   VERIFY: `pytest tests/test_sampler_buffers.py -q`
