# Specification: T-022@20260425T183959Zd

## Problem Statement

`CCS-028` in `my-claw/sdp/prd-cypherclaw-sampler.md` requires a
post-piece sample-usage journal entry per composed piece that records
which samples played, what transformations were used, and what arc
payoff the piece achieved. The current `senseweave.usage_journal`
implementation is only a partial stub:

- it persists `piece_id`, `timestamp`, `samples_played`, and
  `arc_payoff`
- it does not persist the richer sampler-facing fields from the PRD
  (`started_at`, `mode`, `samples_used`, `transformations`, `clicks`)
- it has no real end-to-end write path from the active score-tree
  tracker pipeline
- no integration test currently composes a short piece, compiles it,
  runs tracker scheduling, and asserts that the resulting journal entry
  is actually written with populated sample-usage data

The missing coverage leaves a gap exactly where the sampler PRD expects
observability: future tracker/runtime changes could preserve the
high-level composition tests while silently dropping the journal entry
or emitting one without sample transformations or payoff detail.

## Technical Approach

### 1. Extend the journal model to the CCS-028 contract, but keep it backward-compatible

Update `my-claw/tools/senseweave/usage_journal.py` so the journal entry
can represent both the legacy fields used by the existing tests and the
richer sampler PRD fields:

- keep reading/writing the existing `timestamp` / `samples_played`
  shape so current callers and tests continue to work
- add the richer fields:
  - `started_at`
  - `mode`
  - `samples_used`
  - `transformations`
  - `clicks`
- serialize both legacy aliases and the richer CCS-028 keys from one
  canonical entry object so existing tests continue to round-trip while
  the new integration test can assert the PRD-facing fields directly

### 2. Capture enough tracker metadata to journal real sample usage

The active score-tree tracker path already contains the information
needed for a meaningful journal entry:

- `tracker_compiler.py` writes `sample_gesture_*` metadata into scene
  metadata and sample-lane step metadata
- `music_tracker.py` emits a dedicated `sample` lane when a section has
  sample gesture metadata
- `music_tracker_runtime.py` emits `ScheduledTrackerEvent` objects for
  those sample-lane steps during playback

Thread the per-step and per-scene sample metadata through the runtime
event seam so a post-piece journal helper can record actual emitted
sample events rather than inferring from the planned score alone.

### 3. Add a tracker-runtime journal helper for the end-to-end seam

Add a small helper in `usage_journal.py` that:

- starts a piece tracker for a tracker song / piece id
- records emitted sample-lane events from the runtime callback
- derives the journal entry's ordered `transformations` from the sample
  event metadata (`sample_gesture_transforms`)
- finalizes the piece into a durable JSONL line with:
  - populated sample usage
  - populated transformations
  - populated arc payoff summary
  - populated mode / clicks when provided

This helper should be usable from production code later, but for this
task it mainly provides the public seam the integration test exercises.

### 4. Add a real end-to-end regression test

Create a focused integration test that:

1. commissions a short score-tree piece
2. gates it
3. compiles it through `compile_score_tree_to_tracker(...)`
4. schedules playback with `schedule_song(...)`
5. records emitted sample events into the usage-journal tracker
6. finalizes and writes the journal entry
7. reads the JSONL back from disk and asserts that the written entry has:
   - non-empty sample usage
   - non-empty transformations
   - non-empty arc payoff

The test should exercise the active score-tree tracker path, not the
older freeform `solo_song()` path.

## Edge Cases

- Legacy JSONL entries that only contain `timestamp` /
  `samples_played` must still deserialize correctly.
- Pieces with no sample-lane events must still be writable, but the new
  end-to-end test will force at least one sample-gesture section so the
  populated-path contract is covered.
- Transformations must be derived from actual emitted sample events, not
  from every planned section in the score tree, so automation gating or
  shortened pieces do not over-report unused transforms.
- Only sample-lane runtime events should contribute to the sample-usage
  journal; melody/bass/counter/color events must not pollute the sample
  list.
- `arc_payoff` must still be populated when the entry is written through
  the tracker-runtime hook; an empty payoff string is a failure for the
  new integration path.
- No new dependencies or database migrations are allowed for this task.

## Assumptions

- The score-tree tracker pipeline already routes final score selection
  through `revise_score()`; this task will keep that seam under
  verification but will not redesign the critique flow.
- Resource-governor budget enforcement and startup identity wiring are
  already covered elsewhere; they remain verification anchors rather
  than code-change scope for this journal task.
- `progress.md` is generated and already dirty, so the exploration note
  there remains a single task-line update only.
