# Task frac-0048 Specification: PromptClaw Templates Depth 2

## Problem Statement

`promptclaw/templates.py` owns the starter project scaffold for PromptClaw:
`promptclaw.json`, project guide docs, control prompts, agent prompts, starter
onboarding prompts, memory seed, and example tasks. `promptclaw.bootstrap`
consumes this mapping during `init_project`, and CLI `init` exposes it to users.

The module currently classifies at fractal depth 1 because `project_scaffold`
is the only public function and it directly returns the path-to-content mapping.
The scaffold works, but the module cannot produce meaningful typed output about
the generated template set: callers cannot inspect which paths are startup
prompts, which are docs/control/agent/example files, whether required starter
prompts are present, or the total generated byte count without duplicating the
same file mapping logic.

This task deepens the module to a simple depth-2 implementation by adding one
typed scaffold-entry/report path around the existing output. Existing generated
file paths and contents remain compatible.

## Technical Approach

- Add frozen dataclasses:
  - `ScaffoldTemplateEntry(path, category, content, size_bytes)` for one
    generated scaffold file.
  - `ScaffoldTemplateReport(project_name, file_count, total_size_bytes,
    categories, required_prompt_paths, missing_required_prompt_paths,
    entries)` for an operator-readable scaffold report.
- Add `template_category(path: str) -> str` that classifies scaffold paths into
  one simple category: `config`, `docs`, `control_prompt`, `agent_prompt`,
  `startup_prompt`, `memory`, `example`, or `other`.
- Add `required_startup_prompt_paths() -> tuple[str, ...]` returning the three
  canonical startup prompt paths in deterministic order.
- Add `scaffold_template_entries(project_name: str) -> tuple[ScaffoldTemplateEntry, ...]`
  that builds the same scaffold mapping as `project_scaffold`, then returns
  typed entries in insertion order with UTF-8 byte counts.
- Add `scaffold_template_report(project_name: str) -> ScaffoldTemplateReport`
  that aggregates entries by category, counts total bytes, and reports any
  missing required startup prompt paths.
- Add `summarize_scaffold_templates(project_name: str) -> dict[str, object]`
  that emits a JSON-safe summary with lists/dicts instead of dataclass tuples.
- Keep `project_scaffold(project_name: str) -> dict[str, str]` returning the
  same path/content mapping as before. It can route through the new typed entry
  path, but its output contract must not change.
- Use only the standard library. No new migrations, dependencies, database
  columns, provider secrets, runtime state files, or agent command strings.

## Edge Cases

- Unknown paths classify as `other`.
- Required startup prompts are considered present only when all three canonical
  prompt files are in the generated scaffold mapping.
- Empty project names continue to use the same default configuration behavior
  exposed by `default_project_config`; this task does not add validation.
- `summarize_scaffold_templates` must be directly serializable with
  `json.dumps`.
- `project_scaffold` must preserve all existing generated file paths, JSON
  shape, trailing newlines, and prompt/doc contents.
- Startup identity hardening remains outside this template module, but the
  existing startup tests must continue to cover `bootstrap_identity()` on first
  boot and persistence across standalone/federated flows.

## Acceptance Criteria

1. Existing scaffold/bootstrap behavior remains unchanged.
   VERIFY: `pytest tests/test_bootstrap.py tests/test_config.py tests/test_orchestrator.py -q`

2. `template_category` classifies canonical scaffold paths into meaningful
   template categories and falls back to `other` for unknown paths.
   VERIFY: `pytest tests/test_promptclaw_templates_depth.py::test_template_category_classifies_known_paths -q`

3. `scaffold_template_entries` returns typed entries in scaffold order with
   matching content and positive byte counts for generated files.
   VERIFY: `pytest tests/test_promptclaw_templates_depth.py::test_scaffold_template_entries_preserve_project_scaffold_contract -q`

4. `scaffold_template_report` returns meaningful counts, categories, required
   startup prompt paths, and no missing required prompts for a default scaffold.
   VERIFY: `pytest tests/test_promptclaw_templates_depth.py::test_scaffold_template_report_summarizes_generated_templates -q`

5. `summarize_scaffold_templates` returns a JSON-safe dictionary with file
   counts, category counts, required prompt paths, missing required prompt paths,
   and per-entry rows.
   VERIFY: `pytest tests/test_promptclaw_templates_depth.py::test_summarize_scaffold_templates_is_json_safe -q`

6. Fractal depth for `promptclaw/templates.py` reaches at least depth 2.
   VERIFY: `pytest tests/test_promptclaw_templates_depth.py::test_templates_module_reaches_depth_two -q`

7. Startup identity hardening remains covered for CLI startup and
   standalone/federated first-boot persistence.
   VERIFY: `pytest tests/test_cli_identity_hardening.py tests/test_first_boot.py::TestStartupIdentityPersistence tests/test_governor_integration.py::TestStartupIdentityWiring -q`

8. Full project validation remains clean.
   VERIFY: `pip install -e '.[dev]' && pytest tests/ -x && ruff check src/ tests/ && mypy src/`
