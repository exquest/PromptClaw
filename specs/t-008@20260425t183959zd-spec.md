# Specification: T-008@20260425T183959Zd

## Problem Statement

Subtasks `Za` through `Zc` built the pieces of the auto-tagging pipeline in
`my-claw/tools/sample_capture_daemon.py`:

- `SampleTags` and `extract_context_tags()` normalize capture-time context.
- `analyze_acoustic_features()` measures centroid, bandwidth, and transient density.
- `derive_acoustic_tags()` / `build_sample_tags()` merge those features into
  stable acoustic descriptors.

What is still missing is the write-path integration. Newly captured samples are
not yet tagged when written to disk because there is no persistence helper that
invokes `build_sample_tags()`, writes the audio payload, and stores the emitted
tags in the sample index for later selection/inspection.

This task closes that gap by adding a capture-store/index layer that writes
captured samples and persists both contextual and acoustic tags at save time.

## Technical Approach

1. Extend `my-claw/tools/sample_capture_daemon.py` with a stdlib-only capture
   persistence layer:
   - a capture-store/index helper rooted at a samples directory
   - an `index.sqlite` schema for captured samples
   - a `save_capture(...)` API that calls `build_sample_tags(...)` internally
     before writing the sample
2. Use deterministic, tag-aware filenames:
   - `<source>_<YYYYMMDDTHHMMSSZ>_<arc>_<sampleid>.wav`
3. Persist tag data in two forms:
   - flattened contextual columns (`arc_phase`, `mood_label`, `presence`,
     `time_of_day`, etc.) for direct lookup
   - JSON payloads for `acoustic_tags` and the full tag record so the schema can
     evolve without dropping information
4. Keep the implementation stdlib-only:
   - `sqlite3` for the sample index
   - `wave` for WAV output
   - `uuid` / `json` / `pathlib` for naming and serialization
5. Add tests in `tests/test_sample_capture_daemon.py` that exercise the new
   write path end to end and prove both contextual and acoustic tags are emitted
   from the audio/context inputs rather than hand-supplied metadata.

## Edge Cases

- Missing or partial context state must still write a sample and fall back to
  the existing `SampleTags` defaults (`unknown`, `neutral`, empty acoustic tags).
- Unknown sources should be rejected before any file or index write occurs.
- Repeated writes must reuse the same `index.sqlite` file without dropping
  previously inserted rows.
- Acoustic tags must remain deterministic in storage order so equality and JSON
  assertions are stable.
- The current tree has no detector loop that calls the persistence layer yet, so
  this task defines the write seam that the detector loop can call later.

## Assumptions

- `index.sqlite` is the correct sample-index artifact because the sampler PRD
  already names that file as the library index.
- The write path will initially cover `room` and `contact`, which are the active
  live-capture sources in the current daemon implementation.
- No new dependencies, migrations, or orchestration-doc updates are required.

## Acceptance Criteria

1. Captured samples are auto-tagged at write time by the persistence helper, not
   by caller-supplied precomputed tags.
   VERIFY: `pytest tests/test_sample_capture_daemon.py::test_save_capture_emits_contextual_and_acoustic_tags_for_room_sample -q`

2. Writing a capture persists the emitted contextual and acoustic tags to
   `index.sqlite`.
   VERIFY: `pytest tests/test_sample_capture_daemon.py::test_save_capture_persists_tags_to_sample_index -q`

3. Contact-source captures emit the expected contextual + acoustic tag blend for
   transient-heavy audio and are indexed alongside room captures.
   VERIFY: `pytest tests/test_sample_capture_daemon.py::test_save_capture_indexes_percussive_contact_sample -q`

4. The full sample-capture daemon test file passes with the integrated write
   path in place.
   VERIFY: `pytest tests/test_sample_capture_daemon.py -q`
