# Task T-014@20260502T001347Z: POST /world/entities

## Problem Statement

Deniable's narrative bridge needs to mint new world entities (e.g., a freshly
recruited squad member or a procedurally introduced location) on the
CypherClaw narrative engine over HTTP. T-012 / T-013 added read paths
(`GET /world/entities`, `GET /world/entities/{entity_id}`) but no creation
path exists, so Deniable cannot persist new entities through the narrative
API.

The endpoint must wrap the deploy-side world store rather than reimplement
entity creation, accept `{type, name, domain, properties}`, validate `domain`
against the integration spec's `{shared, cypherclaw, deniable}` whitelist,
and return the created entity with the store-assigned ID so that the same ID
can be re-fetched via `GET /world/entities/{entity_id}` on a follow-up call.

## Technical Approach

- Extend `cypherclaw.narrative_api` rather than adding new packages.
- Add an `EntityCreateRequest` Pydantic schema in
  `src/cypherclaw/narrative_api/schemas.py`:
  - `type`: required non-empty string after stripping.
  - `name`: required non-empty string after stripping.
  - `domain`: required string, validated against `{"shared", "cypherclaw",
    "deniable"}`.
  - `properties`: optional `dict[str, Any]`, defaulting to `{}`.
  - `extra="forbid"` to match the existing request schemas.
- Add a `create_world_entity(...)` helper in
  `src/cypherclaw/narrative_api/entities.py` that mirrors the pattern of
  `append_world_event(...)`:
  - Resolve a creation method via duck-typing across common names
    (`create_entity`, `add_entity`, `record_entity`, `upsert_entity`,
    `save_entity`).
  - Use `inspect.signature` to invoke the resolved method with whichever
    keyword names it accepts (`type`/`entity_type`/`kind`,
    `name`, `domain`, `properties`/`attributes`/`data`, plus a single
    `entity` mapping fallback for stores that take a record dict).
  - Normalize the returned record through the shared `_normalize_entity(...)`
    pipeline so the response carries the deploy-assigned `id`, `type`,
    `name`, `domain`, and `properties`.
  - When the store returns `None` or a value missing fields, fall back to the
    request input for `type`, `name`, `domain`, and `properties` — but never
    fabricate an `id` (an empty `id` from the deploy side surfaces as a 502
    via the route).
- Add a `POST /world/entities` route in
  `src/cypherclaw/narrative_api/app.py`:
  - Use `status_code=status.HTTP_201_CREATED` and `response_model=EntityRecord`,
    matching the `POST /events` shape.
  - Reuse the shared `require_auth` dependency.
  - Map `EventStoreUnavailableError` (and any other store-construction
    exception) to HTTP `503`, matching neighboring endpoints.
  - Wrap the `create_world_entity(...)` call in a try/except mapping to
    HTTP `502` for downstream failures, mirroring the existing
    `append_world_event(...)` route.
  - When the deploy-side path returns an entity record with an empty `id`,
    surface HTTP `502` so the client never receives a creation response that
    is not re-fetchable.
- Do not add migrations or new dependencies. The `entities` table already
  carries `domain` (T-007 series); creation only writes columns that already
  exist.

## Edge Cases

- Missing `type` or whitespace-only `type` is rejected by the schema before
  the store is touched.
- Missing `name` or whitespace-only `name` is rejected by the schema before
  the store is touched.
- A `domain` outside `{"shared", "cypherclaw", "deniable"}` is rejected with
  HTTP `422` before the store is touched. (The PRD §CN-008 explicitly pins
  this whitelist.)
- A request omitting `properties` defaults to an empty dict; the response
  preserves whatever the store persisted, falling back to `{}` if the store
  returned `None`/missing.
- A store implementation returning `None` or the bare new ID still produces
  a fully-populated `EntityRecord` by falling back to the request input —
  except for `id`, which must come from the store. If the store yields no
  ID at all, the route returns HTTP `502`.
- Stores that expose a single-mapping `create_entity({...})` API (record
  dict) are supported via the same `(event/record/entry)` parameter
  inspection pattern used by `append_world_event(...)`.
- Auth behavior matches the rest of the narrative API: when `auth_token` is
  configured, missing/wrong `X-Narrative-Auth` returns `401`; when unset,
  the route is open.
- The auto-generated hardening items about `bootstrap_identity()` / startup
  flow / governor enforcement are unrelated to this HTTP wrapper — the
  narrative API does not own the identity startup path. Existing
  `tests/test_first_boot.py` and `tests/test_governor_integration.py`
  remain verification anchors only.

## Assumptions

- The deploy-side world store exposes (or will expose) one of the duck-typed
  creation method names. This matches the PRD §CN-008 acceptance criterion
  ("New Deniable squad-member entity creates successfully") and the
  existing event-store wrapper pattern. If no creation method is present,
  `create_world_entity(...)` raises `AttributeError` at request time, which
  the route maps to HTTP `502` — surfacing the integration gap to operators
  rather than silently failing.
- The `entities` table's `domain` column already exists (T-007 series), so
  no migration is required for this task.
- Deniable will treat HTTP `502` from this endpoint the same way it treats
  `502` from `POST /events` — degrade and retry on the next narrative cycle
  rather than crash.
