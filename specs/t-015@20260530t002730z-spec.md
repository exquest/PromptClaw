# T-015@20260530T002730Z Spec: Continuous Asset-Bus Producer Run Mode

## Problem Statement

The Deniable Asset Bus producer can process one pending-request snapshot with
`process_pending_requests_once(...)`, but there is no continuous run mode that
keeps polling the bus and handles requests that arrive after the first pass.
Operators need a long-running producer loop that sleeps between polls, then
processes newly arrived request files with the same one-pass idempotency,
dispatch, partial/error classification, and atomic manifest behavior already
covered by T-012 through T-014.

The active ADP process is the task prompt's Explore -> Specify -> Test ->
Implement -> Verify -> Document workflow. No separate standalone ADP document
was found beyond that prompt workflow and prior task specs that mirror it.

## Technical Approach

- Add a typed `run_asset_bus_producer(...)` function in
  `promptclaw.asset_bus.producer`.
- Treat this as DAB-042's programmatic run loop. The PRD's later DAB-052 CLI
  surface (`promptclaw asset-bus run` / `doctor`) remains a separate T1 task.
- Keep `process_pending_requests_once(...)` as the only batch-processing
  primitive. The run mode should call it once per poll rather than duplicating
  request enumeration, dispatch, or manifest classification logic.
- Add a small `ProducerClock` protocol with `sleep(seconds)` and a default
  system clock wrapper around `time.sleep(...)`. Tests can inject a fake clock
  that records sleep calls and writes a request during the interval.
- Add a `ProducerRunResult` dataclass containing the number of completed polls
  and the per-poll `BatchProcessResult` objects. It should expose aggregate
  request-id tuples for processed, partial, failed, and skipped ids so callers
  can inspect bounded test runs without re-parsing batches.
- The default run mode is continuous. A `max_polls` argument bounds the loop for
  tests and smoke commands; `max_polls=None` means run until interrupted or a
  supplied `stop()` predicate returns true.
- Validate loop configuration up front: `poll_interval_s` must be positive, and
  `max_polls`, when supplied, must be a positive integer.
- Export the new run-mode API from `promptclaw.asset_bus`.
- Keep the implementation stdlib-only and avoid changing renderer, runner,
  database, migration, provider-secret, or startup identity behavior.

## Edge Cases

- A request added while the producer is sleeping must be picked up on the next
  poll and processed through `process_pending_requests_once(...)`.
- A first poll with no pending requests should still count as a completed poll;
  the producer then sleeps before the next poll unless it has reached
  `max_polls` or `stop()` returns true.
- `max_polls=1` should perform one immediate poll and return without sleeping.
- A `stop()` predicate that is already true should return a zero-poll result.
- Per-request renderer, dispatch, request-read, and manifest-shape failures
  remain owned by `process_pending_requests_once(...)`; run mode should not
  swallow or reinterpret its per-request summaries.
- Keyboard interrupts remain normal process control and are not caught inside
  the producer loop.
- The recurring SuperCollider `fx_bus_id` and `sw_sampler.scd` hardening
  failures are outside this asset-bus Python slice, but they remain mandatory
  verification anchors.
