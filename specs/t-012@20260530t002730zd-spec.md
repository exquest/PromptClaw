# T-012@20260530T002730Zd Spec: Request Entry-Point Dispatch Wiring

## Problem Statement

The Deniable Asset Bus has a renderer matrix, registry, and `dispatch_request()`
function that route a request's `asset_type` to a named renderer. The
request-processing entry point, `process_request_if_pending(...)`, still takes a
manual render callback and does not read the request file or invoke dispatch.
That leaves the producer unable to exercise end-to-end routing from
`requests/<request_id>.json` through the matrix/registry to the renderer stub
that creates the manifest.

## Technical Approach

- Keep `promptclaw.asset_bus.store.process_request_if_pending(...)` as the
  request-processing entry point because it already owns idempotency and atomic
  result-manifest writes.
- Preserve the existing callback mode so current idempotency tests and callers
  remain compatible.
- Add an explicit dispatch mode: callers provide a `RendererMatrix` and
  `RendererRegistry`; the entry point reads
  `requests/<request_id>.json`, parses JSON into a mapping, invokes
  `dispatch_request(request, matrix, registry)`, and writes the returned
  manifest atomically to `deliverables/<request_id>.result.json`.
- Add a focused integration test that creates two real request files, one
  `image` and one `music`, then verifies each routes to its own renderer stub,
  writes an asset under `deliverables/<request_id>/`, and records the matching
  result manifest.
- Keep renderer stubs hermetic and stdlib-only; no live CypherClaw box, GPU,
  network, provider secret, database migration, or new dependency is involved.
- Treat the recurring SuperCollider `fx_bus_id` and `sw_sampler.scd`
  hardening failures as verification anchors, not as implementation scope for
  this asset-bus Python slice.

## Edge Cases

- Existing result manifest: the entry point remains a no-op and must not read
  the request file or call a renderer.
- Callback mode: existing callers that pass `render=...` continue to work.
- Dispatch mode missing `matrix` or `registry`: reject with a clear
  `ValueError` instead of silently doing nothing.
- Missing or malformed request JSON: surface the file/JSON exception so the
  future producer loop can decide how to write an error manifest.
- Unknown `asset_type` or unregistered renderer: preserve the typed
  `UnknownAssetTypeError` / `UnknownRendererError` behavior from
  `dispatch_request(...)`.
- Renderer returns a mapping-shaped manifest; non-mapping renderer output is
  out of scope for this slice and remains a future producer validation concern.
