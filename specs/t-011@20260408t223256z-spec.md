# Spec: T-011@20260408T223256Z

## Problem Statement

The R750 Ollama integration plan defines a `LOCAL_ONLY` runtime mode that should restrict the live daemon to the local `ollama` agent and prevent cloud-provider CLI/API usage. The current daemon only narrows `_available_agents()` when `LOCAL_ONLY` is set, which leaves multiple cloud paths open: router execution still tries to invoke a cloud CLI, direct `@claude`/`@codex`/`@gemini` dispatch can still request a cloud agent, and plan steps with an explicit cloud agent name are executed as written.

## Technical Approach

1. Keep `_local_only_enabled()` as the single environment check and use it as the daemon-wide feature gate.
2. Add a small daemon helper that normalizes any requested agent to the allowed local agent when `LOCAL_ONLY` is enabled.
3. Apply that normalization at the execution boundary so all agent steps and fallback paths ultimately execute `ollama`, even if an upstream router or manual step requests a cloud agent explicitly.
4. Short-circuit `route_message()` when `LOCAL_ONLY` is enabled so the daemon does not invoke a cloud router CLI; instead it should fall back directly to an `ollama` agent step with the same conversation context.
5. Preserve the normal multi-agent behavior when `LOCAL_ONLY` is unset or false.
6. Update the live-operations docs and changelog to document the new local-only routing mode.

## Edge Cases

- Truthy values should include at least `1`, `true`, `yes`, and `on`; unset or false-like values should preserve normal behavior.
- If `LOCAL_ONLY=true` and a cloud agent is named explicitly in a plan step or direct dispatch, the daemon must execute `ollama` instead of the requested cloud agent.
- If `LOCAL_ONLY=true`, router fallback and quota-error retry logic must not reintroduce cloud agents.
- Existing quota-aware behavior should remain unchanged when `LOCAL_ONLY` is disabled.

## Acceptance Criteria

1. When `LOCAL_ONLY=true`, the daemon available-agent list collapses to `["ollama"]`.
   VERIFY: `pytest tests/test_ollama_run_agent.py::test_available_agents_local_only_restricts_to_ollama -q`

2. When `LOCAL_ONLY=true`, routing does not invoke a cloud router CLI and instead falls back to an `ollama` agent step.
   VERIFY: `pytest tests/test_ollama_run_agent.py::test_route_message_local_only_bypasses_cloud_router -q`

3. When `LOCAL_ONLY=true`, executing a plan step that names a cloud agent still runs `ollama`.
   VERIFY: `pytest tests/test_ollama_run_agent.py::test_execute_plan_local_only_coerces_cloud_agent_step_to_ollama -q`

4. When `LOCAL_ONLY=true`, `run_agent()` does not invoke a cloud subprocess even if the caller asks for a cloud agent explicitly.
   VERIFY: `pytest tests/test_ollama_run_agent.py::test_run_agent_local_only_coerces_explicit_cloud_agent_to_ollama -q`

5. When `LOCAL_ONLY` is unset or false, normal cloud-agent command construction still works.
   VERIFY: `pytest tests/test_ollama_run_agent.py::test_run_agent_cloud_path_unchanged_when_local_only_disabled -q`

6. Existing daemon fallback and quota tests still pass after the local-only guard is added.
   VERIFY: `pytest tests/test_daemon_fallback.py tests/test_agent_selector_quota.py -q`
