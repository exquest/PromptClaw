# Specification: T-021@20260425T183959Za

## Problem Statement

The operator diagnostics surface already reports sampler state, but the current
sample status line collapses the sampler plan and sampler playback into a single
either/or phrase. When `/tmp/sample_dsp_activity.json` says CypherClaw is
currently targeting one source and `/tmp/sample_playback_state.json` says a
rendered sample event is actively playing, operators only see one half of that
story.

This makes it harder to answer two distinct live questions at once:

1. What source is the sampler currently working from?
2. What sample event is actually sounding right now?

The status line should surface both states together as one readable operator
sentence.

## Technical Approach

1. Extend the shared `senseweave.sample_status.sample_status_text(...)` helper
   with an explicit opt-in path for combined plan+playback wording, then enable
   that path from `senseweave.operator_diagnostics` without forcing the face
   display to change its existing sampler phrasing.
2. Treat `/tmp/sample_dsp_activity.json` as the sampler plan/intention source:
   requested source, resolved source, capture readiness, and activity mode.
3. Treat `/tmp/sample_playback_state.json` as the sampler execution source:
   whether a sample event is actively playing, its playback mode, and the
   resolved source that produced it.
4. When both states are present and playback is active, emit a combined
   operator-facing line in this shape:
   `currently sampling <plan-source> · playing sample <mode> from source <source>`
5. Preserve existing degradation behavior:
   missing files, corrupt JSON, inactive playback, or missing fields must fall
   back to the existing plan-only / playback-only / unavailable behavior
   without crashing diagnostics collection.
6. Keep the implementation stdlib-only and avoid changing unrelated diagnostic
   fields.

## Edge Cases

- If playback is active but does not expose a distinct sample identifier, the
  human-readable playback label should use the playback mode because that is the
  only stable descriptive field currently published by
  `sample_playback_state.json`.
- If requested and resolved sources differ, the plan phrase must preserve both
  (`theramini in via room mic`) rather than hiding the retarget.
- If capture is not ready, the status must not falsely claim CypherClaw is
  currently sampling.
- If monitor state reports `no_capture`, the existing monitor-offline signal
  must remain visible.
- If playback is inactive or absent, the status should remain plan-only
  (`currently sampling`, `holding`, or `not ready`) rather than inventing a
  playback clause.

## Assumptions

- No new dependencies or migrations are required.
- The shared sample-status helper is still the correct seam, but the combined
  operator wording must be opt-in so existing face-display wording and its
  integration behavior remain stable unless explicitly changed later.
- The auto-generated hardening items about self-critique wiring, budget
  enforcement, and startup identity sequencing remain out of scope for this
  diagnostics-formatting task and are treated as verification anchors rather
  than implementation targets here.

## Acceptance Criteria

1. When sampler plan and playback state are both present, diagnostics surface a
   single line that includes both the current sampling target and the active
   playback description.
   VERIFY: `pytest tests/test_operator_diagnostics.py::test_status_display_includes_combined_sampler_plan_and_playback -q`

2. The shared sample-status formatter combines plan and playback state into one
   readable line without dropping requested-versus-resolved source information.
   VERIFY: `pytest tests/test_sample_status.py::test_sample_status_text_combines_current_sampling_and_playback_state -q`

3. Missing or corrupt sampler state still degrades cleanly without crashing the
   diagnostics payload.
   VERIFY: `pytest tests/test_operator_diagnostics.py::test_operator_diagnostics_tolerates_missing_and_corrupt_state -q`
