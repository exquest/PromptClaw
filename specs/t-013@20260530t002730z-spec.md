# Task T-013@20260530T002730Z: Asset Bus Producer Batch Pass

## Problem Statement

DAB-040 requires `producer.py` to process every pending Deniable Asset Bus
request found at the beginning of one pass. The existing
`process_request_if_pending(...)` helper correctly handles one request id and
preserves idempotency, but it propagates renderer, dispatch, request-read, or
manifest-shape errors to its caller. A single bad request can therefore abort a
batch before later pending requests receive manifests.

## Technical Approach

- Add `promptclaw/asset_bus/producer.py` as the producer-loop layer above
  `store.process_request_if_pending(...)`.
- Introduce a typed `BatchProcessResult` summary containing processed, failed,
  and skipped request ids for diagnostics and tests.
- Implement `process_pending_requests_once(bus_root, *, matrix, registry)`:
  - call `list_pending_requests(...)` once at the start of the pass;
  - iterate the sorted snapshot of pending ids;
  - call `process_request_if_pending(...)` for each id so existing idempotency,
    request-file loading, dispatch, and atomic manifest write behavior stay the
    source of truth;
  - catch exceptions per request, write a v0.1 `error` manifest to
    `deliverables/<request_id>.result.json`, and continue with the rest of the
    snapshot;
  - leave already-fulfilled ids skipped if they become fulfilled between
    enumeration and processing.
- Export the new producer API from `promptclaw.asset_bus`.
- Keep the implementation stdlib-only and avoid changing renderer, runner, or
  CLI surfaces in this slice.

## Edge Cases

- A renderer or dispatch exception for one request must not prevent later
  pending requests from being processed in the same pass.
- The failing request itself must receive an `error` manifest so it is no longer
  pending forever.
- The error manifest must use the existing v0.1 shape:
  `schema`, `request_id`, `status: "error"`, `producer: "cypherclaw"`,
  empty `assets`, a short `notes` value, and a string `error` reason.
- The pending list is a snapshot. Requests added after enumeration are left for
  a later pass.
- If another process writes a manifest after enumeration, the existing
  idempotency check returns skipped and the producer must not rewrite it.
- No database schema, migration, dependency, provider secret, runtime state
  directory, startup-flow wiring, or agent command is introduced.
- Candidate startup identity hardening is not part of the asset-bus producer
  implementation. It is covered as a regression anchor by the existing startup
  tests plus the CLI startup persistence test added for this task.
