# T-010@20260515T214233Z Spec: Add Slow-Inference Diagnosis CLI

## Problem Statement

PAL-019 needs an operator-facing CLI that diagnoses slow PAL 2026 inference from
bounded read-only evidence. PAL-018 already captures health, saved smoke
baseline token/s, GPU hints, and relevant logs into a context artifact, but
operators still need a direct command that writes a diagnosis run artifact and
does not mutate infrastructure.

## Technical Approach

- Add a `slow_inference_diagnosis` workflow in `promptclaw/pal_agent.py` beside
  the existing PAL triage, action, and slow-inference context workflows.
- Add CLI wiring at `promptclaw pal diagnose slow-inference PROJECT_ROOT`, with
  optional `--task` and `--json`, following existing `argparse` nested PAL
  command patterns.
- Reuse the PAL-018 read-only tool registry:
  - `pal_health`
  - `pal_smoke_baseline`
  - `gpu_hints`
  - `slow_inference_logs`
- Write a normal `.promptclaw/runs/<run-id>/` artifact set:
  - `input/task.md`
  - `routing/route.json`
  - `routing/route.md`
  - `outputs/slow-inference-diagnosis.json`
  - `handoffs/slow-inference-diagnosis.md`
  - `summary/final-summary.md`
  - `logs/events.jsonl`
  - `state.json`
- Keep the workflow local and deterministic. Do not call PAL for a plan or
  summary, do not expose action ids, and record `mutating_actions: []` in the
  diagnosis artifact and route metadata.
- Derive a small diagnosis object from observations: baseline token/s,
  optional log-observed token/s, router health status, optional GPU utilization,
  findings, severity, and operator recommendations.
- Treat missing smoke reports, missing SSH configuration, and unavailable GPU
  hints/logs as diagnosis inputs rather than workflow failures.
- Do not add dependencies, migrations, database columns, secrets, remote write
  commands, or approval-gated actions.

## Edge Cases

- No smoke reports exist: the command still writes a completed diagnosis
  artifact with unavailable baseline and a finding that fresh smoke data is
  needed.
- SSH diagnostics are not configured: GPU/log observations are skipped and the
  diagnosis records that remote evidence is unavailable.
- Router health is not green: the diagnosis includes a router health finding
  without attempting restart or config changes.
- Baseline token/s is poor: the diagnosis marks slow throughput and recommends
  read-only follow-up or explicitly approval-gated remediation, not mutation.
- GPU utilization appears saturated in read-only hints: the diagnosis records
  GPU saturation as evidence and recommends operator review.
- A fixed diagnostic tool raises or returns malformed data: `_run_tool(...)`
  wraps the error so the artifact is still written.
- The generated startup identity hardening bullets target existing startup
  identity code. This task keeps those checks as regression anchors rather than
  expanding the PAL CLI scope into identity rewiring.

   events, and state; the payload and route both record no mutating actions.

   baseline token/s, log-observed throughput regression, and GPU saturation when
   those signals are present.
   VERIFY: `pytest tests/test_pal_agent.py::test_pal_slow_inference_diagnosis_derives_findings_from_context -q`

3. The CLI summary and JSON modes call the diagnosis workflow and expose the run
   id, status, severity, summary path, diagnosis path, executed tools, and empty
   mutating action list.
   VERIFY: `pytest tests/test_pal_agent.py::test_pal_diagnose_slow_inference_cli_prints_summary -q`
