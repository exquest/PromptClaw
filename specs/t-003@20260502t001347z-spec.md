# Task T-003@20260502T001347Z: POST /events

## Problem Statement

Deniable needs to append completed raid outcomes into CypherClaw's shared
narrative event log over HTTP so later narrative reads can observe the outcome
through `GET /events`. The existing narrative API slice already exposes
`GET /events` through an injectable/lazy event-store wrapper. This task adds
the write side for `POST /events` without opening SQLite directly from the
remote consumer.

## Technical Approach

- Extend the existing `cypherclaw.narrative_api` event surface rather than
  introducing a second API package.
- Add a Pydantic request schema for event creation with:
  - `type`: non-empty narrative event type such as `raid.completed`.
  - `domain`: optional domain, defaulting to `shared`.
  - `payload`: JSON object, defaulting to `{}`.
  - `created_at`: optional caller-supplied timestamp string when the upstream
    engine supports it.
- Add a response schema returning the normalized created `EventRecord`.
- Implement `append_world_event(...)` beside `list_world_events(...)`, using
  the same deploy-side event store and result normalization patterns already
  used for `GET /events`.
- Prefer common append method names in order: `append_event`,
  `record_event`, `add_event`, `create_event`, `append`.
- Inspect append method signatures so the wrapper can pass named parameters
  (`event_type`/`type`, `domain`, `payload`, `created_at`) when supported, with
  conservative positional fallback for unknown callables.
- If the deploy-side append method returns `None`, synthesize the response from
  the submitted payload when the in-memory store side effect is already the
  durable authority.
- Return `503` when no event store can be built and `502` when the downstream
  store raises during append.
- Do not add migrations or dependencies in this slice; CN-001 owns database
  domain-column migration separately.

## Edge Cases

- Whitespace-only `type` is invalid and should fail request validation before
  the store is called.
- `payload` must be a JSON object; arrays/strings are rejected by Pydantic.
- Omitted `domain` persists as `shared`, matching the PRD's shared narrative
  default.
- Append methods may name the event type parameter `event_type`, `type`, or
  `kind`; the wrapper should support those names.
- Append methods may return dicts, dataclasses, object-shaped records, or
  `None`; the API normalizes all successful returns into `EventRecord`.
- Downstream append failures should not be hidden behind validation errors.
- Existing `GET /events` tailing and filtering behavior must continue to pass.
- Generated hardening notes about score self-critique, resource-governor
  enforcement, and startup identity are unrelated to this endpoint. They remain
  verification anchors only and should not be bundled into the narrative event
  append implementation.
