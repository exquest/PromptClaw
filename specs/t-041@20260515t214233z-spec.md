# T-041@20260515T214233Z Spec: Approved PAL Deploy-Apply CLI

## Problem Statement

PAL deployment tooling can load the repo-managed manifest, compute deterministic
fake-remote diffs, print a no-write deploy plan, and store changed remote file
content through a backup primitive. Operators still do not have an apply command
that exercises the deployment write path under an explicit approval gate.

This task implements PAL-031 from
`sdp/prd-pal-2026-agentic-ops-platform.md`: fake remote test requires approval
flag. The first apply implementation stays local and deterministic by mutating a
local fake remote inventory snapshot only. It must not contact SSH, restart
services, implement rollback, or infer approval from any other command.

## Technical Approach

- Add `promptclaw pal deploy apply PROJECT_ROOT` under the existing PAL deploy
  CLI group.
- Require `--remote-inventory PATH` so the command has an explicit local fake
  remote snapshot to mutate; no live remote probe is performed.
- Require `--approve-apply` before any fake remote inventory write or backup
  artifact is created. Without that flag, the command returns nonzero and leaves
  inputs untouched.
- Reuse the existing manifest loader, fake remote inventory loader, diff model,
  deploy-plan model, and backup primitive.
- Before overwriting changed fake remote files, store their previous content in
  the local PAL deploy backup artifact.
- Apply only local manifest files that can be written safely:
  - `added` files are written to the fake remote inventory.
  - `changed` files are backed up, then overwritten with local content and
    manifest metadata.
  - `missing` local source files are skipped and reported, not deleted remotely.
  - `unchanged`, unmanaged remote, and excluded runtime files are preserved.
- Emit a JSON-safe result model with `workflow_id`, approval status, fake remote
  path, backup metadata, summary counts, applied entries, skipped entries,
  `remote_writes=true`, `live_ssh=false`, and `service_restarts=false`.
- Keep the implementation stdlib-only with typed function signatures.

## Edge Cases

- Unapproved apply must not create `.promptclaw` directories, backup artifacts,
  or mutate the fake remote inventory snapshot.
- Changed remote files must have content available for backup; otherwise apply
  fails before writing the fake remote inventory.
- Metadata-only changed files still require backup because rollback may need the
  prior mode, owner, or group.
- Optional missing local source files remain skipped; apply must not delete
  remote files in this task.
- Runtime excluded remote paths and unmanaged operator files stay in the fake
  remote inventory output.
- Non-UTF-8 fake remote content must round-trip through JSON safely.
- Startup identity hardening remains an existing regression anchor. This PAL
  apply task does not change startup flow because current CLI, daemon,
  first-boot, and narrative ASGI tests already cover `bootstrap_identity()`
  persistence and ordering before `FirstBootAnnouncer` in standalone and
  federated modes.

## Assumptions

- `--approve-apply` is the explicit approval flag for PAL-031, mirroring the
  future `--approve-rollback` flag named for PAL-033.
- This task's apply command targets a local fake remote inventory snapshot only.
  Live SSH capture/write support remains future work behind the same explicit
  approval boundary.
- Rollback behavior is intentionally out of scope for T-041 and remains owned by
  the later rollback tasks.
