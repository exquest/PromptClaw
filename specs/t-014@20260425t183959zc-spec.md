# Specification: T-014@20260425T183959Zc

## Problem Statement

`CCS-020` now has a repo-backed unit file, boot hook, and a deployed live
service, but the repository still lacks a repeatable operator verifier for the
runtime contract this rollout depends on:

- `cypherclaw-sample-capture.service` must be active under systemd
- the unit must honor its `cypherclaw-jack.service` dependency
- `Restart=always` must recover the daemon after a forced `kill -9`

Without a checked-in verifier, these checks remain an ad hoc shell exercise and
cannot be regression-tested from the repo.

## Technical Approach

1. Add a stdlib-only verification tool at
   `my-claw/tools/sample_capture_service_verify.py`.
2. Make the tool support both:
   - local Linux execution against `systemctl`
   - remote execution via `ssh <host> ...`, so this macOS checkout can verify
     the live `cypherclaw` host without copying scripts around manually
3. Have the tool:
   - assert the sample-capture service is `active/running`
   - assert `After=` or `Requires=` includes `cypherclaw-jack.service`
   - assert `cypherclaw-jack.service` is itself `active/running`
   - record the service `MainPID`, send `kill -9`, then poll until systemd
     restarts the service with a different nonzero PID
4. Add focused regression tests for the tool using fake command runners, matching
   the existing small-tool testing style in this repo.

## Edge Cases

- The development machine may not provide `systemctl`; the verifier must fail
  clearly in local mode and remain usable through `--host cypherclaw`.
- `systemctl show` may return missing or zero PIDs while the unit is restarting;
  the restart poll should tolerate transient `activating` or `failed` states
  until timeout.
- The dependency check should treat either `After=` or `Requires=` as evidence of
  the configured relationship, but the runtime verification should also require
  `cypherclaw-jack.service` to be active before the sample-capture service is
  considered healthy.
- The kill/restart probe must reject a restart that comes back with the same PID
  or never returns to `active/running` within the timeout.

## Assumptions

- The reachable SSH target remains `cypherclaw`, and it has `systemctl`,
  `kill`, and the deployed `cypherclaw-sample-capture.service`.
- No new dependencies or migrations are required; stdlib plus existing system
  tools are sufficient.
- This task does not change the service unit itself unless the verifier reveals
  a real contract gap.

## Acceptance Criteria

1. The repository ships a tested verifier tool for the sample-capture systemd
   runtime contract.
   VERIFY: `pytest tests/test_sample_capture_service_verify.py -q`

2. The verifier can prove `Restart=always` recovery logic from a simulated
   `kill -9` restart sequence in regression tests.
   VERIFY: `pytest tests/test_sample_capture_service_verify.py::test_main_reports_success_after_pid_changes -q`

3. The live CypherClaw host passes the active-state, JACK-dependency, and
   forced-restart recovery checks end to end.
   VERIFY: `python my-claw/tools/sample_capture_service_verify.py --host cypherclaw`
