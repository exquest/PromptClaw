## Instructions

Read the project's ADP documentation and understand existing patterns before making changes.

**ADP Phases:** Explore → Specify → Test → Implement → Verify → Document

### Phase 0: Explore
- Read all files in the affected area of the codebase.
- Identify existing patterns, naming conventions, and architectural decisions.
- Review related tests to understand expected behavior.
- Update progress.md with exploration findings.

### Phase 1: Specify
- Write a specification to `specs/{{ task_id | lower }}-spec.md`.
- Include: problem statement, technical approach, edge cases, acceptance criteria.
- Each acceptance criterion must have a VERIFY command.

### Phase 2: Test Development
- Write tests BEFORE implementation, derived from the specification.
- Tests must FAIL before implementation (Red phase).
- After this phase, test files are LOCKED — do not modify assertions.

### Phase 3: Implement
- Implement the minimum code needed to pass all tests.
- One sub-task at a time, commit each logical unit.
- Type hints on all function signatures.
- Follow existing patterns.

### Phase 4: Verify & Document
- Run full test suite and quality checks.
- Update CHANGELOG.md, progress.md.

## Rules

- No questions — document assumptions in ESCALATIONS.md.
- Do not modify locked test assertions.
- Log new dependencies to ESCALATIONS.md.
- Commit per sub-task: `feat(scope): summary [{{ task_id }}]`

## Common Pitfalls (from recurring verify feedback)

These are the top reasons tasks get PASS WITH NOTES instead of clean PASS.
Avoid them on the first attempt:

- **Always include a migration** when adding or modifying database columns.
  A new model or column without a migration is a deployment gap.
- **Commit all changes** before the verify phase begins. Uncommitted work
  is flagged as blocking by the verifier.
- **Stay in scope** — only commit work under this task's ID. Do not bundle
  changes from adjacent subtasks or refactor unrelated code.
- **Add indexes on new FK columns** in `__table_args__` when adding
  foreign key relationships.
- **Keep CHANGELOG entries consistent** with actual behavior. Do not copy
  descriptions from other tasks or specs without verifying they match
  the implemented code.
